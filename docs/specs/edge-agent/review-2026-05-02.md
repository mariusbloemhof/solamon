## Critical — produces wrong data, breaks an MVP feature, or creates a hard-to-spot bug

### 1. Units gap propagates straight through the poller into the cloud

`docs/specs/edge-agent/modbus-poller.md:80-99` — the poller calls `profile.decode(block.name, raw_bytes)` and writes the resulting values into SQLite verbatim, then the publisher sends them as-is. There is no edge-side units-fix path.

If `architecture/profiles/acuvim_l.yaml` ships without `scale: 0.001` on the power metrics (device-library review #1), every power reading is 1000× off:
- Acuvim returns `12350.0` (W).
- Decoder returns `12350.0` (no scale).
- Cloud catalog says `kW`, range `[-100000, 100000]` — 12350 passes range check.
- Dashboard renders "12350 kW" — wrong by three orders of magnitude.

The edge spec is correct to defer to the profile, but that means the device-library #1 fix is a hard prerequisite for bench Day 4-5 telemetry to be meaningful. Worth flagging here so it doesn't slip while the edge agent is being built.

### 2. `recent_commands` idempotency is in-memory only — restart loses it

`docs/specs/edge-agent/command-subscriber.md:165-178` uses a Python `cachetools.TTLCache` (5-min TTL). After any Pi restart the cache is empty. Combined with QoS-1 broker redelivery (`clean_session=False`, `keepalive=60`), the broker's persistent session can replay an unacknowledged command after the Pi reconnects post-restart — and the Pi will treat it as new and execute the Modbus write a second time.

For `set_value` on `demand_window_minutes` this is benign (idempotent). For post-MVP commands the spec already names — `start`, `stop`, `clear_fault`, `reset_counter` — it is dangerous.

Either:
- Persist processed-command IDs in SQLite (`processed_commands` table with a TTL of `expires_at + grace`).
- Or document explicitly in `command-subscriber.md §5` that idempotency is in-memory only and is safe **only** for value-set commands; non-idempotent command types must wait for the persistent-cache work.

### 3. Replay sweeper references state that doesn't exist

`docs/specs/edge-agent/mqtt-publisher.md:140-152` — the sweeper calls `buffer.unmark_in_flight(...)` on stuck rows. But the buffer schema (`modbus-poller.md:122-134`) only has `published BOOLEAN` (0 or 1) — there is no "in flight" state to unmark. Rows are either `published=false` (eligible for next publish loop iteration) or `published=true`.

If a row stays `published=false` for 60 s, the sweeper's `unmark_in_flight` is a no-op (or an error, depending on implementation). The publish loop will already be picking that row up every 0.5 s anyway.

Either:
- Add an `in_flight_since TIMESTAMP NULL` column; pollers don't touch it; publishers set it when they fetch a batch and clear it after `mark_published`; sweeper resets `in_flight_since IS NOT NULL AND in_flight_since < now() - 60s`.
- Or delete §5 entirely and rely on the publish loop's own retry behaviour, which is what actually does the work.

### 4. Heartbeat `status` enum doesn't include "fault"

`docs/specs/edge-agent/architecture.md:115-119` says: on Modbus fingerprint mismatch the agent halts pollers but keeps heartbeat running, "cloud sees site as 'fault'". But the heartbeat payload only emits `status: "online"` (`mqtt-publisher.md:191`) or `"offline"` (LWT). `"fault"` is never published.

Cloud's `app.device.status` CHECK accepts `online | offline | unreachable | fault | unknown` (`migrations/0001_initial.sql:88-90`) but the cloud spec's `mqtt-contracts.md:5.1` only documents `online | offline` for heartbeat too. Three docs, three different impressions, and "fault" has no path from edge → cloud.

Either add `"fault"` (and a separate event payload to indicate fingerprint mismatch) to the heartbeat schema, or change architecture.md's claim to "cloud sees site as offline (heartbeat suppressed) and a separate `solamon/{slug}/alarms/...` topic carries the fingerprint mismatch event" (which doesn't exist either).

### 5. `control_spec.format_tolerance` is used but not in the profile schema

`docs/specs/edge-agent/command-subscriber.md:144`:

```python
if values_equal(rb_value, value, tolerance=control_spec.format_tolerance):
```

`docs/specs/device-library/profile.schema.json:196-229` `ControlSpec` has no `format_tolerance` field. The schema would reject any profile that declared one.

Either add `format_tolerance` to the schema (with a sane default for floats), or change the edge spec to use a hardcoded tolerance (e.g., float-eps for Float32, exact for ints) without referencing a profile field.

### 6. `validate_control` API contract conflicts between specs

`docs/specs/device-library/profile-schema.md:209`:

```python
def validate_control(self, logical_metric: str, value: Any) -> bool:
    """Validates a control command against allowed_values."""
```

`docs/specs/edge-agent/command-subscriber.md:73-77`:

```python
try:
    profile.validate_control(metric, value)           # checks allowed_values, type
except ValidationError as e:
    ack = build_failed_ack(cmd, status="failed", error=str(e))
```

One says it returns bool; the other expects it to raise. Pick one — raise-on-invalid is more useful (carries the error message) but commits the loader to a Python-specific exception type the cloud also has to import.

### 7. Timestamp UTC-awareness not enforced

The MQTT contract requires explicit UTC offset (`mqtt-contracts.md:48-49`: "RFC 3339 / ISO 8601 with millisecond precision and explicit UTC offset (`Z`)"). The edge spec uses `now().isoformat() + "Z"` (`mqtt-publisher.md:119, 188`).

`datetime.now()` in Python returns a **naive** timezone-unaware datetime by default. Calling `.isoformat()` on a naive value yields `"2026-05-02T12:34:56.789012"` and concatenating `"Z"` produces a malformed string that asserts UTC while actually carrying the Pi's local time. If the Pi happens to be set to local time (Africa/Johannesburg = UTC+2), every telemetry timestamp is two hours into the future.

Use `datetime.now(timezone.utc).isoformat(timespec="milliseconds").replace("+00:00", "Z")` (or document a `now_utc()` helper). Verify with bench rehearsal Day 1.

## Major — gaps that will surface during implementation

### 8. Heartbeat task is specified in two places with different implementations

`architecture.md §4` describes heartbeat as a top-level standalone task in the §2.1 task topology table. `mqtt-publisher.md §7` describes it as a sub-task of the MQTT loop, run inside `asyncio.gather(publish_loop, command_loop, heartbeat_loop)` (line 65) — so when MQTT disconnects, heartbeat_loop dies until reconnect.

The MQTT-loop version is correct (no broker = no heartbeat = LWT covers liveness). The standalone-task version is misleading. Pick one and align both docs.

### 9. Bootstrap token rotation path is unspecified

`config-loader.md §6` mentions "When the cloud rotates the per-site secret (post-MVP feature), the old secret is valid for a 24-hour grace period to allow Pi restart."

But: `bootstrap.yaml.bearer_token` is the bearer used to call `/edge/config/{site_slug}`. After a successful refresh, the response carries the new `mqtt.password`. The new bearer for the **next** call must come from somewhere — but the spec doesn't say the agent overwrites `bootstrap.yaml.bearer_token` after rotation. After 24 h grace expires, the old token in `bootstrap.yaml` fails and the agent can't fetch config.

Either:
- The agent overwrites `bootstrap.yaml.bearer_token` from `mqtt.password` after every successful refresh.
- Or the bootstrap is one-shot and a separate field captures the rotating token in `site_config.yaml`.

Document the rotation flow before the rotation feature is built.

### 10. `../infrastructure/pi-install.md` is a dangling cross-reference

`config-loader.md:32, 152` and `architecture.md` both reference `../infrastructure/pi-install.md`. That folder doesn't exist; only README + 5 sub-specs exist in `edge-agent/`. The bootstrap-file generator, `solamon-edge-install.sh`, has no spec — but config-loader.md depends on it knowing what arguments to pass.

This bites also probe-cli (review #5: "the symlink into `/usr/local/bin`").

Either commit the infrastructure spec or inline the bootstrap-input contract into config-loader.md §2.

### 11. `lib/profile_loader/` has no spec doc and no clear owner

Edge agent (`modbus-poller.md:111`), cloud ingestion (`cloud/ingestion-worker.md`), and probe CLI (`probe-cli/README.md:177`) all import this shared package. Its API is sketched at `device-library/profile-schema.md §7` ("Loader API (Python sketch)") but there's no concrete spec, no schema-validation strategy, no test plan.

Who builds it first? The edge agent can't start polling until it exists. The cloud's seed script can't validate profiles without it. Until someone owns the loader, three specs are blocked on a phantom dependency.

Either: write a `lib/profile-loader/` spec under `docs/specs/`, or fold it explicitly into the device-library spec group as the next deliverable.

### 12. `device.id` change requires manual Pi restart with no automation

The Pi's MQTT subscription is `solamon/{slug}/commands/{device.id}` (`mqtt-publisher.md:61`). `device.id` is the cloud's `app.device.id` UUID, learned at bootstrap. If the cloud reseeds devices (e.g., a profile correction that drops and recreates the row), the Pi keeps subscribing to the old UUID and silently drops every incoming command.

config-refresh runs every 5 min but updates the cache file only — `architecture.md §3.2` and `config-loader.md §4` confirm "in-memory profile is not swapped". So the operator must SSH in and `docker restart`. Document this hazard in §5 of config-loader.md alongside other "operator must restart" cases.

### 13. No re-fingerprint on Modbus reconnect or device replacement

Fingerprint runs once at startup (`architecture.md §3.1 step 7`). If the Modbus device is replaced (e.g., bench unit swapped for prospect-site unit while the agent is up) or its firmware changes mid-life, the agent doesn't notice. It will happily decode `0x0600` against an Acuvim II that returns garbage at that address.

For MVP this is fine — devices don't get hot-swapped — but flag as a known-known. Either: re-fingerprint after every TCP reconnect (cheap, ~5 ms), or document explicitly that device replacement requires `docker restart solamon-edge`.

### 14. `readback_register` has no `fc` field

`device-library/profile.schema.json:212-220` — readback register has `address`, `format`, optional `length`/`offset`, but no `fc`. `command-subscriber.md:125` notes "convention: read with fc=3 regardless" then assigns `readback_fc` to a variable that is never used (line 130 calls `read_holding_registers` directly — fc=3 hardcoded).

Today only Acuvim L's `0x010C` demand window is in scope; FC03 readback is fine. But future profiles may put a writable register in the input-register space (FC04 readback) and the schema can't express it. Add `fc` to readback or document FC03-only as a hard constraint.

### 15. Modbus error counter is per-block in code but per-device in `architecture.md`

`architecture.md §2.2`: `modbus_errors (rolling 60-second window)` — singular, global.
`modbus-poller.md:73, 102`: `metrics.record_modbus_error(block.name)` and `metrics.record_modbus_success(block.name, elapsed_ms)`.

So either: (a) the block name is just for logging and only one global counter is incremented; or (b) per-block counters with a global aggregator. Two interpretations, two implementations. Decide.

This matters because `edge_modbus_errors_per_minute` is a logical metric (`logical_metrics.yaml:533-539`) — what it actually represents is set by this decision.

### 16. "10 consecutive failures → stop polling that block until restart" has no operator signal

`modbus-poller.md §5` (first row): after 10 consecutive Modbus exceptions, the poller halts that block until the next process restart. But:
- No MQTT event published.
- No `device.status` update — heartbeat still says `online`.
- The cloud sees telemetry stop arriving for that block, eventually times out.
- The operator notices via "the demand chart hasn't updated in 30 min".

Either:
- Publish a `solamon/{slug}/alarms/...` event when a block is halted (would require a topic addition — currently not in `mqtt-contracts.md`).
- Or include halted-block names in the heartbeat payload (`halted_blocks: ["energy"]`).
- Or accept the silent halt and document it as known.

### 17. Pi NTP / clock drift not addressed

Edge agent timestamps assume the Pi's clock is correct. If NTP isn't running (or NTP server unreachable in the field), telemetry timestamps drift, and the cloud's hypertable index ordering becomes meaningless.

The Pi install spec would normally cover this, but it doesn't exist (#10). Document in `architecture.md §3.1` that startup verifies `timedatectl` shows a synced clock or fails with `clock_unsynchronised`.

### 18. SQLite buffer write cycle — `INSERT OR IGNORE` masks PK collisions silently

`modbus-poller.md:151` uses `INSERT OR IGNORE`. Combined with PK `(time, device_id, logical_metric_key)`: if two cycles for the same block somehow produce the same millisecond timestamp (clock jump backward via NTP step, or low-precision `now()`), the second cycle's data is silently dropped.

Tighten: either include `block_name` in the PK (today metrics are unique per block, but future overlap could happen) — OR use higher-resolution timestamps (microseconds) — OR log a WARN on `changes() == 0` after the executemany.

### 19. `now().precision` is unspecified

Related to #18: Python's `datetime.now()` has microsecond precision, but `isoformat()` may emit fewer digits on some platforms. The `time TEXT` column in SQLite is ISO 8601 string — collisions depend on rendering precision.

Pin a millisecond format (e.g., `isoformat(timespec="milliseconds")`) and document it.

### 20. atomic_write_cache permissions and ownership

`config-loader.md:78-86` uses `tempfile.mkstemp` then `os.replace`. `mkstemp` creates the temp file with mode 0600 owned by the calling user. `os.replace` preserves source mode. If the agent runs as a non-root `solamon` user but the original file is root-owned (e.g., installer placed it), the rename succeeds but the file is now owned by `solamon` — fine for the agent's reads but might surprise an operator running `sudo cat`.

Minor; document the expected runtime user.

## Minor — doc consistency / polish

### 21. Double-counting of Modbus errors

`modbus-poller.md`: outer `except Exception as e` (line 44-46) calls `metrics.record_modbus_error(block.name)`. Inner `result.isError()` branch (line 71-77) also calls it. A Modbus exception caught at the inner branch is correctly counted once; an exception that escapes the inner branch is **not** caught there (because `isError()` doesn't raise) — so no double count in practice. But the outer except will trigger on connection errors AND the inner branch handles modbus exceptions — different paths, so no double count. Still, the spec reads as if both could fire for one error. Tighten the prose.

### 22. `readback_fc` dead variable

`command-subscriber.md:125` assigns `readback_fc = control_spec.fc` then never uses it (line 130 calls `read_holding_registers` directly). Either remove or actually parameterise the readback FC.

### 23. `unfiltered_messages()` plus topic filter is redundant

`mqtt-publisher.md:60` → `client.unfiltered_messages()` returns all incoming messages. `command-subscriber.md:29` then filters on topic. With a single subscribe (commands topic only), the filter is dead code. Remove or use `messages_for_topic()` directly.

### 24. ASCII control writes can't be expressed in readback path

`command-subscriber.md:130` uses `byte_length(readback_format) // 2` to compute register count. For `format: ascii`, length is per-instance via `metric.length`. The readback path doesn't read it. No ASCII writes are in MVP — flag as known.

### 25. `httpx.AsyncClient(timeout=30.0)` for config fetch

Generous on bench LAN; tight on marginal LTE during a field deploy. Make it `60.0` and document.

### 26. Subscription re-issued on every reconnect

`mqtt-publisher.md:60-61` resubscribes on each connect inside `async with mqtt`. With `clean_session=False`, the broker remembers subscriptions across reconnects — the SUBSCRIBE on reconnect is harmless but adds wire chatter. Some MQTT clients (paho derivatives) re-subscribe automatically. Document either "we always re-subscribe to be defensive" or "asyncio-mqtt does this for us".

### 27. Crash-loop limits at the systemd level assume systemd manages docker.service

`architecture.md §6`: `StartLimitBurst=5, StartLimitIntervalSec=60`. These are unit-file directives — but Docker's restart policy (`restart: always` per `mqtt-publisher.md/architecture.md`) operates inside dockerd. systemd protects dockerd, not the container. To rate-limit container restarts, you need Docker's `restart: on-failure:N` or a custom systemd unit per container.

Pin: either set restart limits in compose, or move the agent out of compose to a systemd-managed unit. Document the chosen mechanism.

### 28. Heartbeat retain=true and LWT will_retain=true are aligned ✓

Confirms the cloud-review #18 fix lands here. No action — note positively for cross-review traceability.

### 29. `parse_command_payload` validation depth unspecified

`command-subscriber.md:33` parses but doesn't say with what (Pydantic against the cloud's command schema would be the natural choice). Pin it.

### 30. `recent_commands` cache size

`TTLCache(maxsize=1000, ttl=300)` — 1000 commands × 5 min = sustained 3.3 cmd/s capacity. MVP nowhere near this; flagged for completeness.

### 31. Buffer ring delete drops unpublished rows older than 7 days

`mqtt-publisher.md:160-175` — explicit trade-off: simpler ring, accepts that an extended outage loses oldest data. Reasonable for MVP. Worth a single line in operator runbook: "if cloud has been offline >5 days, expect data loss; check buffer depth before extending the outage".

### 32. Subscription topic uses `device.id` UUID — what about multi-device?

For SOL-16 (multi-device-per-Pi), the subscription pattern would be `solamon/{slug}/commands/+` with per-message dispatch. Today's hardcoded single device.id is correct for MVP but means the multi-device migration touches this file. Document in OOS section.

### 33. Profile auto-detection deferred — mismatch handling explicit ✓

Architecture §3.1 step 7's "halt poller tasks on fingerprint mismatch" is the right behaviour while SOL-11 isn't built. No action.

### 34. Logging redacts the bearer token but not the MQTT password explicitly

`config-loader.md §6` says the bearer is masked. The MQTT password is the same value (per §2 — "deliberately conflated"). So it's covered transitively. After they diverge (post-mTLS migration), this needs revisiting. Note for the SOL-10 migration.

### 35. The shared `Modbus client` is reused for control writes during pollers

`architecture.md §2.2`: pymodbus serialises operations on a single TCP connection. ✓ Correct. But `command-subscriber.md` doesn't acknowledge that a long-running poll (worst case 5s timeout) will block a control write that arrives mid-poll. For MVP cadence, this is at most ~5 s extra latency. Document the upper bound.

### 36. `now()` for command `received_at` not specified

`command-subscriber.md:190` references `cmd.received_at_iso()`. The parser must inject this at MQTT-message-arrival time. Spec doesn't show. Pin in §2 ("`parse_command_payload` adds `received_at = now()` from the MQTT-message receive event").

## Summary

The single highest risk is **#1 (units gap propagation)** — the edge agent is the place where the device-library #1 bug becomes irreversible (data lands in cloud). Make sure that fix lands before the edge agent first publishes from the bench.

**#2, #3, #4** are all real implementation hazards that surface within the first few sprints: idempotency-on-restart for non-`set_value` commands, replay-sweeper logic that doesn't match the schema, and a heartbeat enum that drops state the rest of the system expects.

**#5 and #6** are contract mismatches with the device-library specs — fix them either at the schema or the consumer side, but commit to one.

**#7 (UTC timestamps)** is a five-line fix that prevents a class of timezone bugs that are hard to spot in the field.

**Major #8-#16** are cross-document inconsistencies and missing operator signals — none block bench Day 4-5, but ignoring them means the spec group has internal contradictions that future contributors will trip over.

**#11 (no profile-loader spec)** is the structural item — the loader is referenced by every implementation track, owned by none, and specified only as a Python-API sketch buried in the schema doc. It needs a home.

Everything else is polish or known-knowns.
