# Probe CLI spec review — 2026-05-02

**Reviewer:** Claude (spec verification duty)
**Scope reviewed:**
- `docs/specs/probe-cli/README.md`

Cross-checked against `docs/specs/device-library/profile-schema.md`, `docs/specs/device-library/acuvim-l-profile.md`, `docs/specs/cloud/`, and `docs/specs/2026-05-02-mvp-design.md`.

The probe-cli leans heavily on the device-library specs (it's a thin CLI over the shared profile loader). Some findings here are downstream consequences of items already raised in the device-library review; they are restated only when they change the probe-cli's behaviour or output contract.

---

## Critical — produces wrong verdicts or blocks an MVP deliverable

### 1. First-match-wins identity probe will mis-identify non-WEB Acuvim II as Acuvim L

`docs/specs/probe-cli/README.md:80-82`:

```
  Trying acuvim_l.yaml: ... Match: acuvim_l (3/3 fingerprint reads passed)
  Trying acuvim_ii.yaml: skipped (acuvim_l already matched)
```

The Acuvim L fingerprint is `expect_exception 0x02 at 0xC350` + positive probes. As noted in device-library review #5, an Acuvim II **without** the AXM-WEB module also returns `0x02` at `0xC350` and (likely) passes the same positive probes — so it satisfies the L fingerprint.

A first-match-wins probe pointed at a non-WEB Acuvim II will declare "ACUVIM L" and silently end up with the wrong block addresses configured on the edge agent. Cross-check would catch the contradiction at the next step, but only because cross-check is forced on for `--verify`; in default mode (per §3.1, "off otherwise" — though §3.2 contradicts this; see #4 below) the operator gets a confident-looking but wrong verdict.

**Fix:** run all profile fingerprints, collect *all* matches, rank by `matched_confidence` (positive_fingerprint > negative_fingerprint), and surface ambiguity to the operator when the top two have the same confidence. The JSON output already has the shape for this (`fingerprint_results` is a map) — just don't short-circuit on first match.

### 2. Bench Day 3 deliverable is overstated — the CLI cannot answer the units question alone

The probe's mandate from `2026-05-02-mvp-design.md §15 OQ#7` and the device-library spec is to "definitively answer the `0x4000` question". The CLI as specified reads both `0x0600` and `0x4000`, decodes per profile, and reports "in range / out of range". It cannot tell whether decoded values are *physically right* — only whether they fall in catalog-permitted ranges.

Linked to device-library review #1: the units gap (Float32 returns W, catalog expects kW). A site reading `12350.0` for `active_power_total` will pass `expected_range: [-100000, 100000]` and report ✓ — and yet be off by 1000×. The probe declares green, the operator commits the profile, billing/dashboards lie.

The bench-Day-3 deliverable is **probe + multimeter cross-check, by a human**. The spec phrasing in §1, §6 implies the CLI is sufficient. Either:
- Split: "the probe verifies decoding succeeds and ranges hold; a separate human step compares against a multimeter for at least one phase".
- Or extend: add a `--reference-value metric=expected_kw` flag so the probe can compare a single decoded metric to an externally measured value.

Without one of these, the bench-Day-3 acceptance criteria at `acuvim-l-profile.md §7` is incomplete.

## Major — gaps that will surface during implementation

### 3. Distribution drift between pip-installed CLI and repo profiles

§5 specifies three distribution targets:

1. **Pi container** — symlink into `/usr/local/bin` "pointing at a script in the agent container".
2. **Pip-install on dev laptop** — `pip install solamon-probe`.
3. **CI fixture** — same source as #2 against a fake Modbus server.

Each target needs `architecture/profiles/*.yaml` and `architecture/logical_metrics.yaml`. The CLI options say `--catalog PATH default: bundled` and `--profiles PATH default: bundled`.

Open questions the spec doesn't answer:
- For pip-install: are the YAMLs included as package data (`MANIFEST.in` / `package_data`)? If yes, *which* version of the YAMLs? An old pip-installed copy on a laptop will probe with stale profiles after the repo updates.
- For the Pi: does the container ship the YAMLs from the repo at build time? Hot-reload is deferred (SOL-17) — does that mean a probe run during a transient hot-edit period sees stale data?
- For CI: the test source pulls from the repo working tree (fine) but pinning expectations against `bundled` means CI needs to point `--catalog` and `--profiles` at the working tree explicitly to avoid surprises.

Pick one story (most likely: package the YAMLs at build, publish a new probe version every time profiles change) and document.

### 4. `--cross-check` default contradicts the §3.2 example

`README.md:48-50`:

> `--cross-check` ... Default: enabled when --verify is set; off otherwise.

`README.md §3.2` (default-mode example, no `--verify`) shows a "Cross-check probe (acuvim_ii.yaml — alternative)" section happening anyway.

Either the default should be "always on" (and the help text is wrong), or the example should be changed to omit cross-check, or `--cross-check` should be on by default with `--no-cross-check` to disable. Right now the implementer can't tell which behaviour to build.

### 5. Pi container symlink isn't executable from the host

§5 distribution method 1: "drops a `solamon-probe` symlink into `/usr/local/bin` pointing at a script in the agent container."

A host-side symlink to a path inside a Docker container is a broken file (the path doesn't exist on the host). What's actually needed is a wrapper script on the host that runs `docker exec solamon-edge solamon-probe "$@"` — or running the probe outside the container entirely with the YAMLs mounted in.

Pick one and document. Otherwise the operator who SSHes in via Tailscale and runs `solamon-probe ...` per §1 finds it doesn't work.

### 6. CI fixture has to simulate Modbus exception 0x02 on `0xC350`

§5 distribution method 3 says CI uses `pymodbus.server.AsyncModbusTcpServer` with "a hand-crafted register file mimicking an Acuvim L". The Acuvim L fingerprint requires `0xC350` to return **exception `0x02` (illegal data address)** — i.e., the address must NOT be in the data block.

`pymodbus` does this by default *if the address is genuinely unmapped*, but the test author needs to know that any sparse register file silently maps reads of out-of-range addresses to either zero (some sparse-store implementations) or raises the exception. Specify the fixture pattern explicitly (probably `ModbusSparseDataBlock` with no entry for `0xC350`).

Likewise the Acuvim II fixture needs the SunSpec sentinel "SunS" at `0xC350` — the test author must construct the bytes correctly. A small fixtures table in the spec would save implementation friction.

### 7. JSON output schema is informal

§3.4 shows an example payload. Programmatic consumers (the post-MVP edge-agent auto-detection, future commissioning scripts) need a schema. Define `matched_confidence` enum values explicitly (`positive_fingerprint | negative_fingerprint | none`?), the shape of `fingerprint_results[*].reads[*]`, and `block_probe[*].metrics_out_of_range[*]`.

Consider committing a `probe-output.schema.json` alongside, mirroring the device-library's profile schema artifact.

### 8. Identifiers are read but not surfaced in the probe report

`device-library/profile-schema.md §4.2` defines `fingerprint.identifiers` (manufacturer / model / firmware / serial ASCII). The Acuvim L profile has `identifiers: []` so the gap doesn't bite today, but Acuvim II profiles will populate it. The probe report templates in §3.2 / §3.3 / §3.4 don't include any output for captured identifiers.

Acuvim II commissioning will want the report to print: `Manufacturer: Accuenergy / Model: Acuvim II / Firmware: H 2.31 / S 3.60 / Serial: ...`.

Add an "Identifiers" section to the human-readable output and an `identifiers` field to the JSON output.

### 9. Profiles with non-Modbus protocols silently break the probe

The device-library schema enum at `profile.schema.json:32-38` permits `connection.protocol: mqtt | http_push`. The probe is Modbus-TCP-only (§7 confirms) but `find_matching_profile` iterates all profiles. What happens when it hits an MQTT profile? Probably an error from the Modbus client. Spec doesn't say.

Cleanest: filter profiles by `connection.protocol == "modbus_tcp"` before fingerprinting and report the others as `skipped: { reason: "non_modbus_protocol" }`. Otherwise, a future MQTT profile breaks every probe run.

(Cross-references cloud review #3 — the MQTT/push discriminator question runs through every spec.)

### 10. Exit code 1 conflates several distinct failure modes

`README.md:60-65`:

```
Exit codes:
  0   ...all probed blocks return sensible values
  1   Identity matches a profile but at least one block fails verification
```

"Block fails verification" can mean:
- Block returned a Modbus exception (block doesn't exist on this device).
- Block read succeeded but values are out of range.
- Transient Modbus timeout.
- Decode failure (e.g., NaN Float32).

The CI test, the operator console output, and the post-MVP auto-detection logic all want to distinguish these. Either split the exit codes (e.g., 5=range-fail, 6=transient) or guarantee the JSON output carries enough detail to act on.

## Minor — doc consistency / polish

### 11. "Single binary" is misleading — it's a Python script

§1: "Single binary, no cloud dependency, no daemon." The CLI is Python (per §4). Distribution targets in §5 confirm this. Just say "single command, no cloud dependency".

### 12. Verbose levels `-v` / `-vv` aren't defined

§3.1 lists `-v` and `-vv` but doesn't say what each adds. At `-vv` (DEBUG) does the operator see the raw Modbus PDU bytes? Decoded Float32 byte order? Just specify briefly.

### 13. No retry on transient Modbus failures during probe

If LTE flaps mid-probe (the operator runs probe over Tailscale-routed Pi → site LAN), one block read might transiently fail. Today that means exit 1 / report a failure. A `--retries N` flag (default maybe 1) would make the probe usable in the field debugging case (§2).

### 14. Sub-variant detection is silently impossible

The Acuvim L family has CL/DL/EL/KL — identical on the wire (`acuvim-l-profile.md §2`). The probe report shows "Verdict: ACUVIM L" but cannot say which sub-variant. The operator must capture this from the device label separately. Acceptance criteria doesn't claim sub-variant detection, but the JSON output also has no slot for "sub_variant: required to be captured externally" — easy to miss in commissioning.

Add a `sub_variant_required: true` flag (or similar advisory field) to the JSON output for profiles where this applies.

### 15. Fast Read mode isn't on the probe's checklist

Per device-library review #15 / `acuvim_l_modbus_registers.md §5`: when AXM-WEB2's Fast Read mode is enabled, all standard FC03 reads to `0x0600` etc. are suspended — but the `0x3000` / `0x3400` Fast Read blocks become accessible. The probe today would just report "block 0x0600 read failed" and exit 1.

Add a Fast Read sanity-check: if `0x0600` errors, also try `0x3000`; if that responds, surface "AXM-WEB2 Fast Read mode appears enabled — disable it via the AXM-WEB2 web UI before re-running the probe".

### 16. Bad port (e.g., 8883) handling not differentiated from Modbus errors

Connecting to a wrong port (e.g., the MQTT TLS port 8883) probably yields garbage bytes or a TLS handshake failure that looks like a Modbus protocol error. Exit code 3 ("connection / Modbus protocol error") covers it but doesn't help the operator who pasted the wrong port. A specific "non-Modbus response" error message would help.

### 17. Connection drops mid-probe not specified

If the probe loses LTE between fingerprint and block-probe, what's the behaviour? Retry from scratch? Partial report? Exit 3 immediately? Spec doesn't say.

### 18. `0V` reading on idle meter passes range check

`expected_range: [0, 1000]` for `voltage_l1_n` includes 0. An idle / dead / disconnected meter happily returns 0 V on every channel and the probe reports green ticks. Soft warning when a phase's V/I is exactly zero would catch wiring or commissioning errors (e.g., the meter has comms but its CTs aren't sensing the load). Optional polish.

### 19. JSON output `matched_profile` shape when nothing matches

§3.4 example is the happy path. What does the JSON look like when `find_matching_profile` returns no matches (exit 2)? `matched_profile: null`? Field absent? Operators / scripts need to know.

### 20. `--quiet` granularity

§3.1 says `--quiet` "suppresses per-block detail". What about per-metric inside blocks? What about identifiers? Probably yes to both, but worth one sentence.

---

## Summary

Probe-cli is a thin spec — most issues are downstream of device-library decisions already flagged. The two material risks are:

1. **#1 (first-match-wins fingerprint)** — the probe will confidently mislabel a non-WEB Acuvim II as Acuvim L. Cheap to fix (run all fingerprints, rank by confidence).
2. **#2 (Bench-Day-3 overstated)** — the probe alone cannot answer the units question; it needs a human-with-multimeter cross-check, and the spec doesn't acknowledge this.

The distribution / packaging story (#3, #5, #6) is the second-biggest implementation hazard — three distribution targets, each with its own way of getting the YAMLs to the CLI, with no single chosen mechanism.

Everything else is polish or known limitations that just need a single sentence each in the spec.
