## Critical — produces wrong data on day one

### 1. Units gap on power metrics (no scale → off by 1000×)

`discovery/specs/acuvim_l_modbus_registers.md:98-138` documents the primary Float32 block as **W / var / VA**. The secondary-block formula (`P = Rx × (PT/CT) [W]`) is consistent — both blocks are W.

The catalog at `architecture/logical_metrics.yaml:128-232` declares `active_power_*` / `reactive_power_*` / `apparent_power_*` in **kW / kvar / kVA**, and `architecture/profiles/acuvim_l.yaml:65-88` maps them with **no `scale`**. Result: a `12350.0 W` reading gets published as `12350 kW`.

The MVP spec example at `docs/specs/2026-05-02-mvp-design.md:194-200` shows `"active_power_total": 12.350` (already kW), so the example contradicts what the profile would actually produce.

**Fix options:**
- Add `scale: 0.001` to all 17 power-unit metrics in `acuvim_l.yaml` (active L1/L2/L3/total, reactive L1/L2/L3/total, apparent total/L1/L2/L3, demand active/reactive/apparent, peak active demand). Currents and PF stay at 1.0.
- Or change catalog units to W / var / VA and rework dashboard.

The bench-Day-3 "open empirical question" at `docs/specs/device-library/acuvim-l-profile.md:193-203` addresses `0x0600` vs `0x4000` but **not units** — add a multimeter-vs-Float32 units check to the same bench session. Note Acuvim II's documented unit *is* kW for the same field (`discovery/specs/acuvim_ii_modbus_registers.md:87-90`), so vendor inconsistency between L and II is plausible.

## Major — gaps that complicate implementation or future profiles

### 2. Vendor-specific format leaked into the global schema enum

`architecture/profiles/profile.schema.json:80` — `acuvim_clock` sits inside `FormatType` alongside generic types. `docs/specs/device-library/profile-schema.md:163-177` (the format-types table) doesn't list it. The next custom decoder (Sinexcel power-flag bitmap, EPC SunSpec model 65534, DSE alarm-bitfield) will grow the enum the same way.

Recommended: introduce a `format: custom` + `decoder: <name>` registry, or accept this as compromise and refactor before profile #2.

### 3. No JSON Schema for the logical metric catalog

`architecture/profiles/profile.schema.json` exists; there is no `architecture/logical_metrics.schema.json`. The catalog structure is documented in a table at `docs/specs/device-library/logical-metric-catalog.md:20-37` but enforced only by code. SOL-14's lint tool can't validate catalog entries on PR.

### 4. Schema cannot express MQTT or HTTP-push profiles

`architecture/profiles/profile.schema.json:32-38` allows `connection.protocol: mqtt | http_push`, but `read_blocks` (`minItems: 1`) and `fingerprint.reads` (`minItems: 1`) are required and FC03-shaped. CLAUDE.md identifies AXM-WEB2 MQTT push as the **primary** billing path — it would not fit. Either narrow the enum to Modbus-only and document push profiles as a separate spec, or split the schema with a discriminator.

### 5. Fingerprint can mis-identify a non-WEB Acuvim II as Acuvim L

`discovery/specs/acuvim_ii_modbus_registers.md:44-45`: the SunSpec `0xC350` block "depends on the AXM-WEB / AXM-WEB2 module being present. Without WEB-family firmware, the block may not be exposed." So an Acuvim II without WEB also returns exception `0x02` → satisfies the negative fingerprint at `docs/specs/device-library/acuvim-l-profile.md:46-49`. Positive probes at `0x0130` (frequency) and `0x0101` (Modbus-addr echo) likely pass on Acuvim II too. The §4 fingerprint-strategy doesn't acknowledge this false-positive case.

### 6. Sub-variant has no home in the data model

`docs/specs/device-library/acuvim-l-profile.md:17-27` says capture `CL`/`DL`/`EL`/`KL` "at commissioning from the device label" and store on the `Device` record. `docs/specs/2026-05-02-mvp-design.md:264` shows `Device` having only `host, port, unit_id, device_type_id`. There is no slot for sub-variant.

Either add `Device.sub_variant` (text), or flag explicitly as a tracked gap. Same applies to firmware version and serial number, which the same paragraph also commits to capturing.

### 7. Orphaned catalog metrics

`voltage_avg_phase`, `voltage_avg_line`, `current_avg`, `load_type` are defined in `architecture/logical_metrics.yaml:70-117, 446` but not mapped by any read block in `acuvim_l.yaml`. Acuvim L's `0x0600` block has no avg registers (II does at `0x4008/0x4010/0x4018`); `load_type` lives in the secondary block at `0x014A`, which we don't poll.

Either drop them, mark as `derived` with a documented computation pattern in the schema, or add the secondary-block read to the profile.

### 8. Demand-window storage cadence is inconsistent across docs

- `docs/specs/2026-05-02-mvp-design.md:135` — "Demand sub-block ... stored at **30 s** (read at 10 s, downsampled)".
- `docs/specs/device-library/profile-schema.md:251` — sub-block sub-cadences are out of scope; "MVP stores at the block cadence; downsampling happens in cloud queries".
- The profile YAML stores demand metrics at the realtime cadence (10 s).

The MVP design table reads as if per-metric storage cadence is supported. Pick one story and align both docs.

## Minor — doc consistency / schema polish

### 9. Catalog top-level field name mismatch

`docs/specs/device-library/logical-metric-catalog.md:154-161` describes "top-level `version: "1.0"` field". The actual YAML uses `schema_version: "1.0"` (`architecture/logical_metrics.yaml:8`). Rename one.

### 10. `acuvim_clock` missing from format-types table

`docs/specs/device-library/profile-schema.md:163-177` lists supported formats; `acuvim_clock` isn't there but is in the schema enum. Either remove (per item #2) or add.

### 11. Schema doesn't conditionally require `format` on fingerprint reads

`architecture/profiles/profile.schema.json:84-110` — `format` is optional on `FingerprintRead`, yet `expected_range` / `expected` / `expected_contains` need it to decode. Add a conditional: `format` required unless `expect_exception` is set.

### 12. Schema doesn't enforce metric-offset alignment

`docs/specs/device-library/profile-schema.md:80-86` says float32 offsets must be multiples of 4, uint16 multiples of 2. The schema currently allows `offset: 1, format: uint16`. Move alignment enforcement into the schema so editor / lint catches it pre-load.

### 13. `MetricMapping.length` unit is undefined

`architecture/profiles/profile.schema.json:184-188` — "Override-byte length, only meaningful for variable-length formats (e.g. ascii)". Bytes or registers? The Acuvim II identifier example uses register count (16 regs). Pick one and document.

### 14. Catalog metric count claim is off

`docs/specs/device-library/logical-metric-catalog.md:165` says "All ~40 metrics above are present". Actual count is ~55. Update.

### 15. Fast Read mode (`0x3000` / `0x3400`) not flagged anywhere

`discovery/specs/acuvim_l_modbus_registers.md:144-168`: when Fast Read is enabled via the AXM-WEB2 web UI, **all other protocols are suspended**. The profile/spec doesn't acknowledge this. Risk: a config change in the AXM-WEB2 UI silently breaks every read. Add commissioning-checklist guidance and a probe-CLI sanity check ("if 0x0600 errors but 0x3000 returns data, Fast Read is on").

### 16. Peak demand is unsigned (`dword_high_first`)

`architecture/profiles/acuvim_l.yaml:124` — peak active power demand decoded as uint32. If a site with significant export ever yields negative peak demand, the value wraps. Acuvim normally reports |max|, so likely a non-issue in practice; flag for awareness.

### 17. `load_type` enum keys are integers in YAML

`architecture/logical_metrics.yaml:452-455` — `enum_values: {76: inductive, 67: capacitive, 82: resistive}`. Valid YAML but JSON keys must be strings; if anything routes the catalog through a JSON Schema validator, integer keys will need stringification. Item #3 (catalog schema) covers this if added.

---

## Summary

The biggest single risk is **#1 (units)** — implementing the profile as currently written will produce numerically wrong telemetry for every power KPI, and it isn't on the bench-test risk list.

The second-biggest is **#4 (schema can't express push profiles)** — this conflicts with the project's stated billing architecture in CLAUDE.md and will block the AXM-WEB2 path the moment billing work starts.

Everything else is tractable polish or known-knowns that should at least be tracked.
