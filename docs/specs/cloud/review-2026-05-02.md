## Critical — produces wrong data, breaks an architectural commitment, or blocks an MVP feature

### 1. `timeseries.reading.value` can only store numeric values — but the catalog has enum, datetime, and string metrics

`docs/specs/cloud/migrations/0001_initial.sql:226` declares `value DOUBLE PRECISION`. There is no string / datetime column.

`docs/specs/cloud/ingestion-worker.md:99-101` says:
> For enum metrics: `value` is the string symbolic form OR the integer code; coerce to the canonical form (string).
> For datetime metrics: `value` is an RFC 3339 string; parse → store as datetime.

`architecture/logical_metrics.yaml` defines:
- `meter_clock` — `data_type: datetime`
- `load_type` — `data_type: enum`
- `voltage_wiring_type`, `current_wiring_type` — `data_type: enum`

`docs/specs/cloud/mqtt-contracts.md:107-114` allows `string | null` in `readings` for exactly these cases.

The MQTT contract permits strings; the worker spec promises to store them; the SQL has nowhere to put them. Implementation will either drop non-numeric metrics silently or crash on insert.

**Fix options:**
- Add `value_text TEXT` and/or `value_time TIMESTAMPTZ` columns to `timeseries.reading`, NULL for unused.
- Or move non-numeric metrics to a separate table (`timeseries.reading_event` or similar).
- Or move `meter_clock` / wiring types out of the hypertable entirely and store on `device_snapshot` only.

`app.logical_metric.data_type` already enumerates `bool, enum, bitfield, datetime, string` — the schema commits to all of these but the storage doesn't accommodate them.

### 2. Cloud → Pi command publish has no retry on MQTT failure

`docs/specs/cloud/control-relay.md:90-106`: the `MqttError` branch logs and continues. Status stays at `pending`. The TTL reaper marks the row `expired` after 5 minutes with no further attempt to publish.

The operator gets `202 Accepted` for a command that may never reach the Pi. The web UI sees `pending` for 5 minutes, then `expired`, with no actionable signal that "the broker was unreachable; retry now".

Either:
- Add a publish-retry task (exponential backoff up to `expires_at`).
- Or surface the publish failure on the resource (e.g., a `last_publish_error` column) so the UI can prompt re-issue immediately.

### 3. `source: "mqtt_push"` reservation conflicts with the AXM-WEB2 architecture

`docs/specs/cloud/mqtt-contracts.md:86` reserves `source: "mqtt_push"` for "AXM-WEB2-driven Acuvim L push, post-MVP".

Per `CLAUDE.md`, the AXM-WEB2 push is the **primary billing channel** — it connects to Mosquitto with **its own credentials**, publishes its **own JSON shape** (`{gateway, device, readings: [{name, value, unit}]}`), to a **configurable topic**, with no concept of the solamon `block` field.

The current envelope (`solamon/{site_slug}/telemetry/{device_id}` + the §4.1 schema) cannot host AXM-WEB2 messages without a transformation step. The `source` placeholder will never be reachable on the topic it lives in.

Either:
- Spec the AXM-WEB2 path as a separate topic + payload schema in this same doc.
- Or remove the `mqtt_push` enum value until that work lands and the architecture is concrete.

### 4. Per-site heartbeat handler updates **all** devices at the site to one status

`docs/specs/cloud/ingestion-worker.md:154-158`:

```sql
UPDATE app.device SET status = $status WHERE site_id = (SELECT id FROM app.site WHERE slug = $slug);
```

Heartbeats are per-Pi, not per-device. With one device per Pi (current MVP) this is fine. The moment SOL-16 (multi-device-per-Pi) lands, **every** device at a site gets marked online when only one is reporting, and every device gets marked offline when only one fails.

The heartbeat payload (`mqtt-contracts.md:5.1`) has no `device_id` either. Decide now whether heartbeats are per-Pi (and `device.status` is updated by Modbus-success signals, not heartbeats) or per-device (and the payload needs `device_ids` array). Today's spec is internally inconsistent for the next-but-one milestone.

### 5. `/health` path mismatch between API spec and OpenAPI

`docs/specs/cloud/api-surface.md:200`:
```
GET    /health                                            → unauthenticated
GET    /api/v1/sites/{slug}/health                        → authenticated
```

`docs/specs/cloud/openapi.yaml:23-26` sets the server URL at `/api/v1`, then declares `paths: /health` and `paths: /sites/{slug}/health`. So both paths are auto-prefixed to `/api/v1/...` — the bare `/health` for liveness probes does not exist in the OpenAPI artifact.

Caddy's reverse-proxy rules will need to know which is which. Either move `/health` to a separate OpenAPI document (or out of OpenAPI entirely) and document the Caddy route, or update the spec to use `/api/v1/health` everywhere.

## Major — gaps that will surface during implementation

### 6. WebSocket endpoints are absent from the OpenAPI artifact

`docs/specs/cloud/api-surface.md:206-259` defines `/api/v1/ws/sites/{slug}/devices/{device_id}/live` and `/api/v1/ws/commands/{id}/live`. `docs/specs/cloud/openapi.yaml` has no entry for either.

OpenAPI 3.1 doesn't natively spec WebSocket, but the typed-SDK story for the web UI relies on the OpenAPI handshake. Either:
- Add an AsyncAPI doc for the WS surface alongside `openapi.yaml`.
- Or accept that WS message types are hand-typed in the web UI and document this trade-off.

### 7. Snapshot upsert WHERE-clause loses same-millisecond writes

`docs/specs/cloud/ingestion-worker.md:138-145`:

```sql
ON CONFLICT (device_id) DO UPDATE
  ...
  WHERE EXCLUDED.snapshot_time > app.device_snapshot.snapshot_time;
```

If two messages arrive with identical `snapshot_time` (different blocks published in the same millisecond, or two blocks read in lock-step), the second update is silently skipped — the snapshot will not include the second message's metrics.

Use `>=` and rely on the JSONB merge to combine. Or split snapshots per block.

### 8. JWT in WebSocket query string leaks to access logs

`docs/specs/cloud/api-surface.md:209-210` permits the JWT via `?token=` query string. Caddy access logs, browser history, proxy logs, and external monitoring will all capture full URLs, including the token.

Use `Sec-WebSocket-Protocol` only, or document this as known and disable query-string-token in production via config.

### 9. Edge bootstrap is chicken-egg on `site_slug`

`docs/specs/cloud/api-surface.md:170-194`: the Pi calls `GET /api/v1/edge/config/{site_slug}` to learn everything. But `site_slug` itself isn't returned — it's the URL parameter. Where does the Pi get its `site_slug` and the matching MQTT password before the first call?

Implied answer: provisioned at install time by `solamon-edge-install.sh` (per `2026-05-02-mvp-design.md:101`). That script's spec doesn't exist yet. Add a one-paragraph "bootstrap inputs" section to `api-surface.md §4.6` and link to the install-script spec.

### 10. Snapshot timestamp ambiguity makes "last seen" misleading

`app.device_snapshot.snapshot_time` is set to the most recent message's timestamp (`docs/specs/cloud/ingestion-worker.md:139-145`). When `realtime` (10 s) and `energy` (60 s) blocks both feed the snapshot, the snapshot_time tracks `realtime` — and energy values up to 60 s old are labeled with the newer time.

Per-metric `last_updated_at` inside the JSONB, or per-block snapshot rows, would be more honest. For MVP just document that `snapshot_time` is "most-recent message time, not most-recent value time".

### 11. `pg_dump` of TimescaleDB hypertables needs care

`docs/specs/cloud/data-model.md:172-178` says "Daily `pg_dump` to S3 in `af-south-1` via cron in a backup sidecar container."

Standard `pg_dump` produces a dump that requires the Timescale extension on restore and benefits from Timescale's `pg_dump -Fc` recommendations. Restoring a hypertable requires extension creation and chunk re-registration semantics that vanilla `pg_restore` doesn't know about.

The bench-rehearsal "restore tested" check on Day 11-12 will catch this if it's not addressed earlier — but the spec should at least name the Timescale-recommended dump approach.

### 12. No retention/cleanup mechanism for `app.dead_letter`, `app.audit_entry`, or `app.control_command`

`docs/specs/cloud/data-model.md:144-149` table:
- Audit log: indefinite (no policy needed).
- Control commands: 3 years.
- Dead letter: implicitly mentioned as "retained for 30 days" in the SQL comment at `0001_initial.sql:205`, but no DELETE / pg_cron / app-level sweep is specified.

Add a periodic cleanup spec (or pg_cron entry) for `dead_letter` and `control_command`, or accept indefinite growth.

### 13. Stored MQTT password is plaintext and lands in `pg_dump` output

`app.site.mqtt_password TEXT NOT NULL` (`0001_initial.sql:53`). Backups are unencrypted by default. The data-model spec acknowledges plaintext storage with "DB-level access is admin-only" — but doesn't extend the same posture to backups in S3.

Either: encrypt the column in MVP (it's just a per-site shared secret — `pgcrypto`'s `pgp_sym_*` works), or document that S3 backup buckets need the same access controls and an encryption-at-rest policy.

### 14. `app.device_type.profile_yaml` JSONB silently loses hex-address notation

YAML hex addresses (`address: 0x0600`) parse to integers. JSONB stores integers. Round-trips (`GET /api/v1/device-profiles/...` → editor → re-store) lose the original hex notation forever.

For MVP this only affects `architecture/profiles/acuvim_l.yaml` → DB → `EdgeConfig` response. The edge agent reads integers and works fine. Future profile-editor UI (SOL-12) needs to handle this. Document it.

### 15. `logical_metric.enum_values` integer-keyed map breaks JSON

`architecture/logical_metrics.yaml:452-455` has `enum_values: {76: inductive, 67: capacitive, 82: resistive}` — integer keys. JSON keys must be strings; on insert into `app.logical_metric.enum_values JSONB`, keys are silently cast to `"76"` etc. The seed script and any caller comparing integer Modbus reads against the JSONB map must coerce types. (Already item #17 in the device-library review — leaving here so the cloud spec carries the awareness.)

### 16. Timezone column has no validation

`app.site.timezone TEXT NOT NULL DEFAULT 'Africa/Johannesburg'` — no CHECK constraint against `pg_timezone_names`. A typo on site creation produces silent date-bucketing bugs. Validate at app layer (Pydantic) and/or add a CHECK against `pg_timezone_names`.

### 17. Late ack arriving after `expired` produces silent state divergence

`docs/specs/cloud/control-relay.md:131-132`:

> if cmd.status in ("confirmed", "failed", "expired"): log.warn(...); return

If a command was marked `expired` cloud-side but the Pi actually performed the Modbus write before the ack got through (slow LTE), the ack is dropped and the cloud believes the device state hasn't changed. The UI shows "expired"; reality is "applied". Decide:
- Should a late ack post-`expired` revert to `confirmed`?
- Or raise an operator alert ("device may have applied an expired command")?

Currently neither is specified.

### 18. Mosquitto LWT retain flag not specified

`docs/specs/cloud/mqtt-contracts.md §5` retains `heartbeat` (`true`) and configures LWT but does not specify whether the LWT message is published with `will_retain: true`.

Without `will_retain: true`, after a Pi crashes:
- The last retained "online" message stays on the topic.
- The non-retained "offline" LWT delivers to currently-connected subscribers only.
- A fresh dashboard subscriber sees "online" forever.

Add `will_retain: true` to §10's connection parameters table, or explicitly state the desired semantics.

### 19. `POST /commands` returns 202 before MQTT publish state is known

Same issue as #2 viewed from the API: the resource state on the 202 response can be `pending` (publish failed) or `sent` (publish succeeded). The web UI has to inspect the returned resource to know which it got — but the spec doesn't say this. Distinguish in the response or always block on publish before responding.

## Minor — doc consistency / polish

### 20. `app."user"` table name requires quoting on every reference

`user` is a Postgres reserved word. Every query needs `app."user"`. Convention: rename to `app.users` or `app.app_user`.

### 21. `timeseries.reading` has no `received_at` column

The spec at `data-model.md:114` distinguishes "edge-agent timestamp (NOT cloud `received_at`)" but the table never stores cloud-receive time. Useful for end-to-end-latency dashboards and clock-skew debugging. Add a nullable column.

### 22. `WebSocket` envelope inconsistency

`/api/v1/ws/sites/.../live` uses `{type: reading|snapshot|heartbeat, data}`.
`/api/v1/ws/commands/{id}/live` uses `{type: current|transition, data}`.
Both use `{type, data}` — ✓ aligned shape, but the union of types isn't documented anywhere as a discriminator. Add a shared envelope schema to `openapi.yaml` or AsyncAPI alongside.

### 23. `/sites/{slug}/devices/{device_id}/readings` 30-day max range vs. 90-day retention

`api-surface.md:108` caps raw queries at 30 days; `data-model.md:146` retains 90 days. Either align or document why the API limit is tighter (probably memory/perf, but it's not stated).

### 24. `EdgeConfig.mqtt.password` field not annotated as sensitive in OpenAPI

`openapi.yaml:246` — `password: { type: string }`. Add `description: "Sensitive — never log"` or use `format: password` for the SDK code-gen hint.

### 25. `EdgeConfig` payload likely benefits from gzip

Profile YAML + catalog YAML rendered as JSON is ~30 KB per response. Caddy can compress; spec doesn't require it. Add a one-line note that Caddy compresses application/json responses.

### 26. Mosquitto ACL regeneration job is unspecified

`mqtt-contracts.md:251` — "Generated and rewritten by the cloud at deploy time from the active list of `app.site` rows. Mosquitto reloads the ACL file on `SIGHUP`."

No script, no trigger, no failure mode. When a site is added in the DB, who actually regenerates the ACL and SIGHUPs Mosquitto? Belongs in this spec or in the infrastructure spec (SOL-21) cross-reference.

### 27. Migration tracking has no checksum

`app.schema_migrations` records `(filename, applied_at)`. No checksum. If `0001_initial.sql` is edited after apply, nothing detects the drift. Acceptable for MVP; flag for the Alembic migration.

### 28. State diagram inconsistency between main spec and `control-relay.md`

`docs/specs/2026-05-02-mvp-design.md §10` shows `acknowledged` as a deliberate intermediate state. `control-relay.md:212` says "`acknowledged` is optional — most flows skip it and go straight to `confirmed`/`failed`". The web UI rendering will see the column populated only sometimes. Either drop `acknowledged` from the lifecycle, or commit to it.

### 29. `device_snapshot.metrics` JSONB merge is shallow

The merge `metrics || EXCLUDED.metrics` (`ingestion-worker.md:144`) only handles flat keys → values. Today metrics are flat ✓, but the JSONB type's looseness invites future shape drift. CHECK constraint or Pydantic validation on the way in would help.

### 30. `organisation.billing_contact_email` is unused in MVP

Defined in `0001_initial.sql:34`. No endpoint serves or sets it. Acceptable column-for-the-future; flag for the next migration to either populate or drop.

### 31. No documented index on `app.audit_entry(actor_id, time DESC)`

Common audit-list query "what did user X do" will table-scan once the audit table grows. Add an index in 0001 or as a separate migration when the audit-browser UI ships.

### 32. `/api/v1/auth/logout` with expired/missing JWT

OpenAPI declares `bearerAuth` required → 401 on missing token. But spec at `api-surface.md:51` calls logout a "no-op". Decide: 401 if unauthenticated, or 204 always. Currently the spec disagrees with the OpenAPI artifact.

### 33. `/api/v1/auth/me` Edge bootstrap auth scheme reuses MQTT password

The Pi authenticates to `/edge/config/{site_slug}` using the **same** secret it then receives in the response. Practically harmless (you already had it) but documented ambiguously — clarify that the response's `mqtt.password` is *the same* as the bearer token, and that the only reason it's there is so the Pi can write it to its config file alongside the rest.

---

## Summary

The biggest single risk is **#1 (value-column type mismatch)** — the moment ingestion encounters `meter_clock` or `load_type`, it either drops the data or crashes. This affects the very first end-to-end test on bench Day 4-5.

**#2 (no command-publish retry)** and **#19 (202 response before publish state known)** are the same root cause and combine into a poor operator experience for the marquee MVP control demo. Worth fixing together.

**#3 (mqtt_push reservation)** is structurally significant — the placeholder field encodes a flawed mental model of the AXM-WEB2 path. Fix it now, before the billing spec is written against a wrong assumption.

**#4 (per-site heartbeat update)** is fine for MVP but a one-commit landmine for the next sprint (SOL-16). Decide the heartbeat semantic now.

Everything else is tractable polish or known-knowns that should at least be tracked.
