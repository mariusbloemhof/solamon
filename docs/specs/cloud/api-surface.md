# Cloud — API surface

**Spec group:** [cloud](README.md)
**Linear:** [SOL-8](https://linear.app/solamon/issue/SOL-8)
**Machine artifact:** [`openapi.yaml`](openapi.yaml) (skeletal; filled in during implementation)

The FastAPI HTTP and WebSocket surface. Auth, REST endpoints, live-data WebSockets, error response convention. The Next.js web app and the Pi edge agent both consume this surface; the OpenAPI spec is the contract that drives the TypeScript SDK generation on the web side.

---

## 1. Stack

| Concern | Choice |
|---------|--------|
| Framework | **FastAPI** (Python 3.12+) |
| Server | `uvicorn` with one worker (single-process MVP) |
| Validation / serialisation | **Pydantic v2** |
| ORM | **SQLAlchemy 2.x async** + `asyncpg` driver |
| Auth | **JWT** issued by FastAPI; bcrypt for password storage |
| OpenAPI | Auto-generated by FastAPI from route signatures + Pydantic models |
| WebSocket | FastAPI's native `WebSocket` |
| MQTT client (in-process) | **`asyncio-mqtt`** (a paho-mqtt asyncio wrapper) |

## 2. URL prefix and versioning

All HTTP endpoints under `/api/v1/`. The `v1` prefix is reserved for the MVP API; future breaking changes get `/api/v2/`. Within a major version, additions are additive (Pydantic models accept unknown-field-tolerant config; clients ignore unknown response fields).

WebSocket endpoints under `/api/v1/ws/...`.

## 3. Auth

### 3.1 Login

```
POST /api/v1/auth/login
Request:  { "email": "admin@solamon", "password": "..." }
Response: { "access_token": "<jwt>", "token_type": "bearer", "expires_in": 86400, "user": { ... } }
```

JWT signed with HS256. Secret is `SOLAMON_JWT_SECRET` env var (32-byte random; rotated on credential rotation). Lifetime 24 h. No refresh token in MVP — re-login.

### 3.2 Logout

```
POST /api/v1/auth/logout
Request:  (empty; bearer token in header)
Response: 204
```

For MVP this is a no-op response (JWT is stateless; logout is a client-side token discard). Post-MVP we add a JWT denylist for explicit revocation.

### 3.3 Identity

```
GET /api/v1/auth/me
Response: { "id": "...", "email": "...", "display_name": "...", "tier": "operations", "role": "admin" }
```

Used by Next.js to populate the user menu and gate route access.

### 3.4 Edge agent auth

The Pi authenticates to **Mosquitto**, not to this REST API, via the per-site MQTT username/password. **Except** for the bootstrap call: `GET /edge/config/{site_slug}` accepts a bearer token equal to the per-site MQTT password (i.e., the same secret authenticates both the MQTT connection and the config-fetch HTTP call). This keeps the Pi to a single secret in MVP.

## 4. Endpoint catalogue

Pagination convention: `?limit=<n>&cursor=<opaque>`. Default limit 50, max 500. Cursor is a base64-encoded opaque string (in MVP, just `time:<rfc3339>`; can become richer later).

Error response convention (RFC 7807 problem+json):

```json
{
  "type": "https://solamon.example/errors/validation",
  "title": "Validation failed",
  "status": 422,
  "detail": "Field 'window_minutes' must be one of [1, 5, 10, 15, 30]",
  "instance": "/api/v1/sites/johansworkbench/devices/.../commands"
}
```

### 4.1 Auth (3 endpoints — see §3)

```
POST   /api/v1/auth/login
POST   /api/v1/auth/logout
GET    /api/v1/auth/me
```

### 4.2 Sites & devices

```
GET    /api/v1/sites                                      → SiteSummary[]
GET    /api/v1/sites/{slug}                               → SiteDetail (includes nested devices, snapshot times, last heartbeat)
GET    /api/v1/sites/{slug}/devices                       → DeviceSummary[]
GET    /api/v1/sites/{slug}/devices/{device_id}           → DeviceDetail (config, profile metadata, snapshot)
```

### 4.3 Telemetry

```
GET    /api/v1/sites/{slug}/devices/{device_id}/snapshot
       → { snapshot_time, metrics: { logical_metric_key: value, ... }, operating_state, active_faults }

GET    /api/v1/sites/{slug}/devices/{device_id}/readings
       Query params:
         metric=active_power_total            (required; can repeat for multi-metric)
         from=2026-05-02T00:00:00Z            (required)
         to=2026-05-02T12:00:00Z              (required; max range 30 days for raw)
         aggregate=raw|1min|15min|1hour       (default raw)
       Response:
         {
           "metric": "active_power_total",
           "aggregate": "raw",
           "points": [ { "time": "...", "value": 12.35, "quality": "good" }, ... ],
           "next_cursor": "..."
         }
```

For MVP, only `aggregate=raw` is supported. Continuous aggregates (1-min, 15-min, 1-hour) are post-MVP; the API accepts the param now so the front end can be implemented once and the back end fills in later.

### 4.4 Control

```
POST   /api/v1/sites/{slug}/devices/{device_id}/commands
       Body: { "logical_metric": "demand_window_minutes", "type": "set_value", "parameters": { "value": 30 } }
       Response: 202 Accepted
                 Location: /api/v1/commands/{id}
                 Body: ControlCommand resource (status=pending or sent)

GET    /api/v1/sites/{slug}/devices/{device_id}/commands?limit=50
       → ControlCommand[] (paginated, recent first)

GET    /api/v1/commands/{id}
       → ControlCommand (full state, including audit_entries[])
```

`ControlCommand` resource shape:

```json
{
  "id": "01952b7c-3a14-...",
  "device_id": "...",
  "logical_metric": "demand_window_minutes",
  "type": "set_value",
  "parameters": { "value": 30 },
  "status": "confirmed",
  "issued_at": "...",
  "sent_at": "...",
  "acknowledged_at": "...",
  "confirmed_value": { "value": 30 },
  "error_message": null,
  "expires_at": "...",
  "issued_by": { "id": "...", "display_name": "Marius" }
}
```

### 4.5 Catalog and metadata

```
GET    /api/v1/metrics-catalog                            → LogicalMetric[]
GET    /api/v1/device-profiles                            → DeviceProfile[] (just metadata + profile_yaml; not full)
GET    /api/v1/device-profiles/{profile_slug}             → DeviceProfile (full, including profile_yaml)
```

These are read by the Next.js admin pages for "browse profiles" (read-only in MVP — editor is [SOL-12](https://linear.app/solamon/issue/SOL-12)).

### 4.6 Edge-facing config

```
GET    /api/v1/edge/config/{site_slug}
       Auth: Bearer <site mqtt password>
       Response:
         {
           "site": { "slug": "...", "id": "...", "timezone": "...", "name": "..." },
           "device": {
             "id": "...",
             "host": "...",
             "port": 502,
             "unit_id": 1,
             "profile_slug": "acuvim_l"
           },
           "profile": { ...full profile YAML rendered as JSON... },
           "logical_metrics_catalog": { ...full catalog... },
           "mqtt": {
             "broker_host": "cloud.solamon.example",
             "broker_port": 8883,
             "username": "solamon-johansworkbench",
             "password": "<the same secret used for this auth>",
             "client_id": "solamon-edge-johansworkbench",
             "topic_prefix": "solamon/johansworkbench"
           }
         }
```

The Pi calls this once at startup, caches the response in `/etc/solamon/site_config.yaml`, and uses it as the source of truth for everything. If the call fails, the Pi falls back to the cached copy.

### 4.7 Health

```
GET    /health                                            → { "status": "ok", "db": "ok", "mqtt": "ok" }
GET    /api/v1/sites/{slug}/health                        → SiteHealth
```

`/health` is unauthenticated for liveness probes. `/api/v1/sites/{slug}/health` requires auth and reports per-site Pi heartbeat freshness, last-reading age, Modbus error rate, buffer depth.

## 5. WebSocket endpoints

### 5.1 Live device stream — `/api/v1/ws/sites/{slug}/devices/{device_id}/live`

After authentication (JWT in `Sec-WebSocket-Protocol` header or `?token=` query string), the server subscribes the connection to live telemetry and snapshot updates for the device.

**Server → client messages:**

```json
{
  "type": "reading",
  "data": {
    "time": "2026-05-02T12:34:56.789Z",
    "metric": "active_power_total",
    "value": 12.35,
    "quality": "good"
  }
}
```

```json
{
  "type": "snapshot",
  "data": { "snapshot_time": "...", "metrics": { "...": "..." } }
}
```

```json
{
  "type": "heartbeat",
  "data": { "status": "online", "last_modbus_success": "...", "buffer_depth_seconds": 0 }
}
```

**Client → server messages:** none in MVP. Subscriptions are inferred from the URL.

### 5.2 Live command stream — `/api/v1/ws/commands/{id}/live`

Subscribes to lifecycle transitions for a single command. Used by the control panel UI to render the live state machine.

**Server → client messages:**

```json
{
  "type": "transition",
  "data": {
    "status": "confirmed",
    "at": "2026-05-02T14:32:02.760Z",
    "confirmed_value": { "value": 30 }
  }
}
```

Connection auto-closes when the command reaches a terminal state (`confirmed`, `failed`, `expired`).

## 6. CORS

For MVP the Next.js web UI is served from the same origin as the API (Caddy reverse-proxies both). CORS allowlist is empty. If/when we host the web UI on Vercel or similar, we add the origins to a `SOLAMON_ALLOWED_ORIGINS` env var.

## 7. Rate limiting

None in MVP. Add when traffic justifies (or when we expose endpoints to a third party). The cloud is single-tenant so abuse is not a concern.

## 8. OpenAPI generation

FastAPI auto-generates OpenAPI 3.1 at `/api/v1/openapi.json` (and Swagger UI at `/api/v1/docs`, ReDoc at `/api/v1/redoc`). The skeletal `openapi.yaml` in this folder is the **starting handshake** — the implementer expands it as routes are built; the live API publishes the authoritative version.

The Next.js client generates a typed TypeScript SDK from the live `/openapi.json` at build time using `openapi-typescript`. This keeps the client and server schemas in lockstep.

## 9. Acceptance criteria

- Every endpoint listed above is implemented in FastAPI; appears in `/api/v1/openapi.json`.
- All Pydantic request/response models validate without `extra="forbid"` on input (so future minor-version additions don't break old clients) but `extra="forbid"` on output (so we don't accidentally leak fields).
- `/health` returns 200 with all components healthy on a fresh deploy.
- The Next.js TypeScript SDK generated from the running API compiles cleanly against the web-ui specs (cross-validated in the web-ui spec group).
- Auth: an expired JWT returns 401; a malformed JWT returns 401; an unauthorised cross-tenant access returns 403.
- Pagination cursors round-trip — fetching `?cursor=X` returns the next page; fetching `?cursor=Y` from that response returns the page after.
