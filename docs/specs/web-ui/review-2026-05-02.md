## Critical — produces wrong UI behaviour or breaks an MVP feature

### 1. WebSocket reading-message shape mismatch — example accesses fields at the wrong level

`docs/specs/web-ui/live-data.md:110-114` (PowerHeroTile example):

```typescript
onReading: (msg) => {
  if (msg.metric === "active_power_total") {
    queryClient.setQueryData(["snapshot", device_id], (prev) => ({
      ...prev,
      metrics: { ...prev.metrics, active_power_total: msg.value },
      snapshot_time: msg.time,
    }));
  }
},
```

But `docs/specs/cloud/api-surface.md:213-224` defines the WS reading envelope as:

```json
{
  "type": "reading",
  "data": {
    "time": "...",
    "metric": "...",
    "value": ...,
    "quality": "..."
  }
}
```

And `live-data.md:158-164` itself shows the dispatcher unpacking by `msg.type` and passing the **whole envelope** to `onReading`. So inside the handler, `msg.metric` and `msg.value` are `undefined`; the access should be `msg.data.metric` / `msg.data.value` / `msg.data.time`.

Implementation built from this example will silently fail every `if` check and never update card values. The error is just an example today, but examples are what the implementer copies.

### 2. Two contradictory implementations of `useDeviceLiveStream`

`live-data.md §4` shows the hook opening a fresh `new WebSocket(...)` inside `useEffect` for every caller — one connection per `useDeviceLiveStream` call.

`live-data.md §4.3` ("Multiple subscribers, one connection") then says "the hook implements **subscription multiplexing**: ... a singleton WebSocket pool keyed by `device_id`, with reference counting." — i.e., one connection across all callers.

These are different implementations. The dashboard has 12+ cards subscribing to the same device; the §4 implementation opens 12 sockets, the §4.3 implementation opens 1. Pick one and align both sub-sections — and align the acceptance criterion at `live-data.md:284` ("Subscribing to the same device from multiple cards opens exactly **one** WebSocket connection"), which assumes §4.3.

### 3. `<EdgeHealthCard>` expects `halted_blocks` from heartbeat that the edge agent doesn't publish

`components.md §3`:

> `<EdgeHealthCard>` ... Modbus errors / minute, last successful read (relative time), Pi heartbeat (relative time), buffer depth in seconds, **halted blocks if any**.

But `docs/specs/cloud/mqtt-contracts.md §5.1` and `docs/specs/edge-agent/mqtt-publisher.md §7` heartbeat payloads have no `halted_blocks` field. The edge-agent review #16 already flagged this as an unsignalled state on the edge side.

So when a Modbus block hits the "10 consecutive failures → stop polling" condition (`edge-agent/modbus-poller.md §5`), the dashboard card cannot show it. The data the web-ui spec assumes simply doesn't exist on the wire.

Either:
- Add `halted_blocks: string[]` to the heartbeat payload schema in `mqtt-contracts.md` (and surface it in the edge agent).
- Or remove the field from `<EdgeHealthCard>` and document the gap.

### 4. `<DemandTile>` peak timestamp can't be served — collides with cloud value-column type bug

`components.md`:

> `<DemandTile>` ... Current `active_power_demand` + peak with timestamp.

`requirements/acuvim_mvp_register_scope.md §J`:

> "Peak this billing period: 287 kW @ 2026-05-08 14:23"

The catalog declares `active_power_demand_max_timestamp` with `data_type: datetime` (`architecture/logical_metrics.yaml:374-379`). Per cloud review #1, `timeseries.reading.value` is `DOUBLE PRECISION` only — datetime values can't be stored. The metric exists in the catalog but is unreachable through the storage path.

So the tile's "with timestamp" promise renders `null` until the cloud schema fix lands. Either flag explicitly here as "blocked by cloud #1" or design a fallback for when the timestamp is absent.

### 5. `last_publish_error` field is invented; the `503` path doesn't exist in the cloud spec

`live-data.md:221-227`:

> The 202 response carries the persisted `ControlCommand` (with `status='sent'` on the happy path or `'pending'` with `last_publish_error` if the broker had transient trouble — see cloud spec §3.4).
> ...
> The `503 Service Unavailable` path renders a more prominent error toast: "Cloud broker unreachable; ..."

Both inventions:

- **`last_publish_error`** is not a column on `app.control_command` (`migrations/0001_initial.sql:151-167`) and is not a field on the `ControlCommand` OpenAPI schema (`openapi.yaml:175-196`). Cloud review #19 flagged that the cloud should distinguish "publish succeeded" vs "publish failed" in the response — but no cloud spec actually does it yet.
- **503 from `POST /commands`** is also not in `api-surface.md §4.4` — only 202 and 422 are documented.
- The "see cloud spec §3.4" cross-reference is broken: control-relay.md only goes to §3.3, and api-surface.md §3.4 is about edge-agent auth, not commands.

The fix-direction is correct (UX needs to show "broker unreachable, retry"), but the web-ui spec has jumped ahead of the cloud spec. Either align the cloud spec to add 503 + `last_publish_error`, or remove this section from web-ui until the contract exists.

### 6. Power values rendered with `formatKw(value)` regardless of whether the value is in W or kW

`live-data.md:125`:

```tsx
return <Metric value={formatKw(value)} sparkline={...} />;
```

This is downstream of device-library review #1 (Float32 returns W; catalog says kW; no scale in profile). If the upstream profile fix doesn't land, `formatKw(12350)` renders "12,350 kW" — three orders of magnitude wrong on a 12 kW load.

Web-ui can't fix this on its own — the profile must scale W → kW before the cloud stores. Worth flagging here as a release-blocker dependency.

## Major — gaps that will surface during implementation

### 7. WebSocket auth mechanism is asserted in web-ui but not yet in the cloud spec

`live-data.md §4.2`:

> Per [`../cloud/api-surface.md`](../cloud/api-surface.md) §5.1: JWT goes in the `Sec-WebSocket-Protocol` header **only** — the `?token=` query string variant was removed because URLs leak to logs.

But `docs/specs/cloud/api-surface.md:209-210` still says:

> After authentication (JWT in `Sec-WebSocket-Protocol` header **or `?token=` query string**), the server subscribes...

The web-ui has implemented the cloud-review #8 fix, but the cloud spec hasn't been updated. Either align the cloud spec or remove the "removed" claim.

### 8. `apiServer` has two different signatures in two specs

`auth.md §3.1`:

```typescript
export async function apiServer<T>(path: string, init?: RequestInit): Promise<T>
```

A generic fetch wrapper.

`live-data.md §2`:

```typescript
const [site, snapshot] = await Promise.all([
  apiServer(session).getSite(slug),
  apiServer(session).getDeviceSnapshot(slug, FIRST_DEVICE_ID),
]);
```

A factory returning an SDK-style object with named methods.

These can't both be right. Pick one (probably the SDK style is what `openapi-typescript` generates) and update both files.

### 9. Pagination component (page-based) is mismatched with API (cursor-based)

`components.md §4`: `<Pagination>` — shadcn-based pager.

`api-surface.md §4`: pagination is `?limit=<n>&cursor=<opaque>` — cursor-based.

shadcn's stock pagination component expects total page counts and integer page numbers. Cursor-based pagination has no "page 5 of 12" concept; just "next" / "previous". Either:
- Use a "Load more" button instead of `<Pagination>`.
- Or wrap shadcn's pager in a cursor-aware adapter (and document the page-number semantics).

The audit list at `/sites/{slug}/control` and `/sites/{slug}/devices/{device_id}` will hit this immediately.

### 10. Snapshot WS message handling replaces instead of merges

`live-data.md:119-122`:

```typescript
onSnapshot: (msg) => {
  queryClient.setQueryData(["snapshot", device_id], msg.data);
},
```

The cloud's snapshot upsert (per cloud spec ingestion-worker.md §4.3) merges incoming metrics into the existing snapshot. The WS forwards the *current full snapshot* — but only after the merge, so `msg.data` should be the post-merge complete state. ✓ if the cloud sends the full state.

But: if the cloud only forwards delta updates per block, then this replaces wipes other-block metrics from the cache. The contract isn't pinned anywhere — `api-surface.md §5.1` shows the snapshot envelope but doesn't say whether it's full state or partial.

Pin the contract on the cloud side (full state, always) or change the web-ui handler to merge.

### 11. `Sec-WebSocket-Protocol` auth still leaks JWTs to access logs

`live-data.md §4.2` correctly flags that query-string tokens leak to logs. But `Sec-WebSocket-Protocol` headers are also captured by Caddy access logs by default — Caddy logs request headers unless explicitly redacted via the `delete` directive in the format block.

Either:
- Document that the Caddy config must redact `Sec-WebSocket-Protocol` from access logs.
- Or accept that the WS auth mechanism still leaks tokens to backend logs and treat backend-log access as part of the secret's access boundary.

### 12. Multi-tab / multi-operator control panel race not addressed

`pages.md §6`:

> "if the operator clicks Apply while a previous command is still pending, disable the button until the WS receives a terminal state."

Single-tab UI race handled. But: two operators, two tabs, both issue commands. Cloud accepts both (no command-level lock). Pi processes them sequentially. Result: a wider race than the UI prevents.

For MVP with one shared admin user this is unlikely, but document the bound: "the Pi processes commands serially; concurrent issuance from multiple sessions results in last-write-wins on the device". Not a UI-side fix; just a documented limitation.

### 13. XSS would steal the access token via `session.accessToken`

`auth.md §2` callbacks make `session.accessToken` available client-side via `useSession()`. Required for `apiClient` and for the WebSocket bearer. But: any successful XSS (e.g., a Tremor-rendered tooltip that reflects unsanitised input from a future feature) reads it and can call any API endpoint as the user.

The standard mitigations (HttpOnly cookies for auth) are incompatible with the Sec-WebSocket-Protocol auth mechanism — the JS *needs* the token. Acceptable for MVP single-user with no untrusted input paths; flag explicitly in `auth.md §7`.

### 14. Server-side validation of `Sec-WebSocket-Protocol` not specified in cloud spec

`auth.md §3.3`:

> Server validates the second protocol string as a JWT and either accepts (echoing back `solamon-bearer`) or rejects with close code 1008.

But `cloud/api-surface.md §5` doesn't specify the protocol-string format, parsing rules, or close-code semantics. The web-ui spec writes against a contract that the cloud spec hasn't committed to. Pin in cloud spec.

### 15. `react-syntax-highlighter` pulled in for a single read-only page

`pages.md §9`: profile detail uses `react-syntax-highlighter`. That package is ~150 kB gzipped (with one language theme). For a single admin page on a < 250 kB MVP bundle (per `live-data.md §7`), it's a substantial chunk.

Either:
- Lazy-load that route's bundle (Next.js does this by default if it's a separate page — verify with bundle analysis).
- Or use `<pre>` with a tiny manual highlighter (or none).

### 16. `<RelativeTime>` per-instance auto-update will run 30+ intervals on the dashboard

`components.md §4`: `<RelativeTime>` is "auto-updating client component". Implies a `setInterval` per instance. The dashboard has many cards each potentially showing relative time → 30+ intervals.

Aggregate via a single context-broadcast tick to avoid the timer churn. Minor performance polish.

### 17. PT/CT enum integer-keyed values may render as integers

`<SiteSetupSheet>` shows `current_wiring_type` (enum). The catalog's `enum_values` map has integer keys (`{0: "3CT", 1: "2CT", 2: "1CT"}`). After JSONB round-trip the keys become strings, but the *value* the snapshot carries for that metric is an integer code (e.g., `0`).

If the API returns `metrics: { current_wiring_type: 0 }`, the dashboard needs to map `0 → "3CT"`. The web-ui spec doesn't say whether this mapping happens client-side (looking up `enum_values` from the catalog) or whether the API pre-resolves enums to their string form.

Pin: either the cloud spec's snapshot endpoint resolves enums (and `data_type: enum` always serialises as a string), or the web-ui owns the lookup.

### 18. Status pill colour mapping not enumerated

`components.md §1` lists `badge` for "Status pills (online / offline / fault)". The full enum is `online | offline | unreachable | fault | unknown`. Design tokens (§5) define four colours: `ok | warn | crit | muted`. Web-ui doesn't say which status maps to which colour.

A natural mapping would be: online=ok, fault=crit, unreachable=warn, offline=muted, unknown=muted. Spec it.

### 19. `<PowerFactorCard>` mixed-sign warning is a heuristic, not a CT-flip diagnostic

`components.md`:

> "Mixed signs across phases are highlighted with a warning border (CT polarity flipped diagnostic)."

Acuvim L's PF sign indicates leading vs lagging on the **phase**, not just CT polarity. A heavily-reactive load (a large motor with capacitor banks on one phase) will produce mixed-sign PF without any CT polarity issue. The diagnostic will false-positive.

Either:
- Reword the warning ("Mixed PF signs — check load characteristics or CT polarity").
- Or pair the PF check with a per-phase active-power sign check (CT-flipped phases also typically show negative active power on a load-only site).

## Minor — doc consistency / polish

### 20. NextAuth v5 (Auth.js v5) maturity

`auth.md §1`. Auth.js v5 was in beta/RC for a long stretch. As of mid-2026 it should be stable; verify the chosen version is GA and not behind a `next` tag. Trivial.

### 21. No login rate limiting

`auth.md` doesn't address brute force on `/login`. Cloud spec also doesn't (cloud review #6). Single shared admin password means an attacker who phishes the URL could brute-force at HTTP latency. Acceptable for MVP single-tenant; flag in `auth.md §8`.

### 22. WebSocket close code semantics not enumerated

`live-data.md §4.1` describes auto-reconnect on `onclose` but doesn't enumerate which close codes are auth-failures (1008), which are normal closures (1000), which are abnormal (1006), and how each affects backoff. Easy spec addition.

### 23. SDK method naming not pinned

Examples like `apiClient.postCommand` and `apiClient.getDeviceSnapshot` assume a particular naming output from `openapi-typescript`. Generated names depend on operationId in the OpenAPI spec — `openapi.yaml` doesn't set operationId on most paths, so the generator produces either anonymous or path-derived names. Either set operationId in the OpenAPI artifact, or pin a code-gen config.

### 24. `<DataFreshness>` and `live-data.md §6.1` describe the same behaviour

Components.md lists `<DataFreshness>` ("Last update: 3s ago") and live-data.md §6.1 says "Every live card has an internal `lastUpdate` state". They're the same thing. Cross-reference one from the other and avoid documenting twice.

### 25. 200 ms first-paint target is aspirational

`live-data.md §8` acceptance: "Initial dashboard render shows real numeric values within 200 ms of nav (server-rendered)". Includes initial fetch RTT to FastAPI + render + network to browser. Plausible on bench LAN; tight on a public deployment over slower links. Either soften the budget or document the measurement context (bench, local Caddy).

### 26. `API_BASE` and `WS_BASE` configuration not specified

Referenced throughout. Probably env vars (`NEXT_PUBLIC_API_BASE`, `NEXT_PUBLIC_WS_BASE`) but never named. Add a one-line "configuration via env" section.

### 27. Bundle budget likely tight with Tremor + Recharts

`live-data.md §7`: < 250 kB initial. Tremor 3.x + Recharts is ~180-220 kB gzipped on its own; shadcn primitives + Next.js runtime add another 100+. Real number is more like 350-450 kB gzipped initially. Either revise the budget or commit to a code-split/lazy-loading strategy in the spec.

### 28. WebSocket reconnect does not re-issue session token

`live-data.md §4.1` reconnects on close. The `useEffect` dependency array (§4) is `[device_id, session?.token]` — so a token rotation does trigger a reconnect ✓. But within an in-flight reconnect cycle, the closure still has the old token. Document explicitly that NextAuth's session refresh will trigger a re-render and the hook will reconnect with the fresh token.

### 29. Clock-skew expiry check

`auth.md §6`: client-side `Date.now() > session.expiresAt` proactive check. Wrong client clock → spurious or skipped redirects. Trivia for MVP; one sentence in §6 acknowledging the assumption.

### 30. Empty-state for `/sites/{slug}` with no readings ever

`pages.md §12.3` lists empty states but not "fresh site with no telemetry yet". Cards would render with `null`/`undefined` snapshot values. Spec a "waiting for first reading..." state.

### 31. AppShell's `<SiteSelector>` tone inconsistency

`pages.md §2` describes the selector as "links to other sites". `components.md §4` describes it as "renders disabled with site name" for MVP. Both true (current state vs. eventual behaviour); consolidate.

### 32. `<Pagination>` component would be unused if cursor-based "Load more" replaces it

If item #9 is resolved by switching to "Load more", remove `<Pagination>` from the components inventory.

### 33. `react-window` for metric grid virtualisation

`pages.md §7`: device detail's metric grid renders 30+ mini-charts. "If not, virtualise via `react-window`." Better to commit upfront — 30+ chart instances are guaranteed perf hazards on Pi-class operator devices. Just plan to virtualise.

### 34. Profile YAML round-trip rendering caveat

`pages.md §9`: "Hex addresses rendered as hex (not the integer JSONB representation — render-time conversion)". Good. But: not every integer field is an address. The renderer needs to know which fields to format as hex (`address`, `readback_register.address`) — a flat "render every integer as hex" would mangle `length`, `cadence_s`, `offset`, etc. Pin the field allowlist.

### 35. No spec for the OpenAPI-generated SDK's build pipeline

Both `live-data.md` and `auth.md` reference the SDK. Build-time generation step (`openapi-typescript` against the running cloud or against the committed `openapi.yaml`) isn't specified. Implementation detail but worth a one-paragraph "build pipeline" section in `live-data.md` so the implementer knows when the SDK is regenerated.

## Summary

The most impactful single fix is **#1 (WS message shape mismatch)** — every dashboard card built from the documented example will silently fail to update. It's a 2-character fix in the example (`msg.metric` → `msg.data.metric`) but copy-paste replication of the wrong shape is the kind of error that hides until live tests.

**#2 (two contradictory `useDeviceLiveStream` implementations)** is a sub-spec internal contradiction — pick the multiplexing one to match the acceptance criterion.

**#3, #4, #5, #6** are all consequences of upstream specs that haven't quite caught up: the heartbeat doesn't carry `halted_blocks`, the `Reading.value` column can't store datetimes, `last_publish_error` and 503 don't exist in the cloud spec yet, and the units gap is unfixed. The web-ui has been written against the *intended* state of those upstream specs but not the *current* state. Worth resolving by running another upstream pass to land the dependent fixes — or by pinning explicit "depends on cloud-fix-X" markers in the web-ui spec.

**#7 (WS auth mechanism)** is the same story for cloud review #8 — fix is already in web-ui, hasn't propagated to the cloud spec.

**#8 (`apiServer` signature)**, **#9 (pagination mismatch)**, **#17 (enum rendering)** are the implementation hazards that bite first on day-zero of dashboard development.

Everything else is polish or known-knowns.
