## Critical — `solamon-cloud-bootstrap.sh` cannot reach the documented happy path

### 1. Mosquitto cert volume mount points to a host path that is empty by design

`scripts/solamon-cloud-bootstrap.sh:249`:

```yaml
- ./caddy/data/caddy/certificates/acme-v02.api.letsencrypt.org-directory/${DOMAIN}:/mosquitto/certs:ro
```

But Caddy's data volume is declared as a **named Docker volume** at line 264 (`caddy-data:/data`), not a host bind mount. Caddy writes its certs into the named volume `caddy-data`, which Docker stores at `/var/lib/docker/volumes/solamon_caddy-data/_data/...`. The host path `./caddy/data/caddy/...` referenced in the Mosquitto mount is **never written to** — Docker auto-creates it as an empty directory the first time the Mosquitto container starts.

Mosquitto loads `cert.pem` from `/mosquitto/certs/`, finds an empty directory, fails to start TLS, fails to start the listener, exits or restart-loops.

Fix options:
- Make Caddy's data a host bind mount (`./caddy/data:/data`) so the path predictably exists.
- Or use a shared named volume both containers mount (Caddy writes, Mosquitto reads).
- Or revert to the `caddy-and-dns.md §6` MVP-pragmatic choice: Mosquitto runs its own quarterly certbot cron, no cert-share at all.

### 2. Even if the path were right, the Caddy filenames don't match Mosquitto's config

`mosquitto.conf` (cloud-bootstrap.sh:205-207):

```
cafile /mosquitto/certs/chain.pem
certfile /mosquitto/certs/cert.pem
keyfile /mosquitto/certs/privkey.pem
```

Caddy stores certs under `<domain>/<domain>.crt` and `<domain>/<domain>.key` — not `cert.pem` / `privkey.pem`. There is no chain file separate from the cert in Caddy's storage layout.

Even with #1 resolved, Mosquitto would 404 on every cert file.

### 3. `cafile` enables mTLS client-cert verification — incompatible with username/password auth

Mosquitto's `cafile` directive tells the broker to **require connecting clients to present a TLS client certificate signed by this CA**. With `cafile chain.pem`, Mosquitto only accepts clients whose cert chains to Let's Encrypt — but the Pi connects with username/password (per `cloud/mqtt-contracts.md §10`), no client cert. Every connection is rejected.

Fix: remove the `cafile` line entirely. Server TLS doesn't need it; mTLS is post-MVP (planned with the move to AWS IoT Core per main spec §13).

### 4. Mosquitto password file is empty; cloud cannot authenticate to its own broker

`scripts/solamon-cloud-bootstrap.sh:216` creates `/opt/solamon/mosquitto/passwords` as an empty file via `touch`. The "first-site procedure" at `cloud-bootstrap.md §10` says it's populated when sites are added — but the **cloud's own** `solamon-cloud` user is also missing from the file.

Then the `app` container's env is configured at line 277:

```yaml
- MQTT_USERNAME=solamon-cloud
- MQTT_PASSWORD=${DB_PASSWORD}
```

(a) `${DB_PASSWORD}` is the Postgres password being reused as the cloud's MQTT password — they should be separate secrets.
(b) Whatever the value is, Mosquitto has no `solamon-cloud` user defined to authenticate it.

The cloud's `mqtt: ok` health probe (per `cloud/api-surface.md §4.7`) fails. The bootstrap script's verification at line 358-371 exits 2.

Fix: generate a `MQTT_CLOUD_PASSWORD` separately, run `mosquitto_passwd` inside the Mosquitto container against the password file to create the `solamon-cloud` user, and reload Mosquitto. Repeat for each site as it's added.

### 5. Mosquitto password file permissions block the in-container Mosquitto user from reading

Line 217: `chmod 600 /opt/solamon/mosquitto/passwords` — owner-only readable, owned by `root` on the host.

The Mosquitto container (`eclipse-mosquitto:2`) runs as UID 1883 (`mosquitto` user) by default. The host file mode 600 owned by root is unreadable by UID 1883 inside the container. Mosquitto refuses to start or runs without authentication.

Fix: `chmod 0644` (or 0640 with explicit group ownership) and document the UID assumption, OR populate the file from inside the running Mosquitto container using `mosquitto_passwd`, OR run Mosquitto as root inside the container via `user: root` in compose (not recommended).

### 6. Mosquitto `depends_on: caddy: service_started` doesn't wait for cert provisioning

Line 251-253: Mosquitto waits for Caddy to *start*, not for Caddy to *finish provisioning* TLS certs from Let's Encrypt. Cert provisioning takes 10-30 s after Caddy starts. During that window, Mosquitto starts, tries to load certs, fails, restarts.

`restart: unless-stopped` means Mosquitto restart-loops until certs appear — eventually OK, but ugly and prevents bench Day-1 acceptance from being clean.

Fix: this is moot if items #1-#3 are resolved by certbot or shared volume. If staying with the cert-share approach, add a healthcheck on Caddy that returns healthy only after certs are provisioned.

### 7. Migration step assumes `psql` is installed in the FastAPI app image

`scripts/solamon-cloud-bootstrap.sh:341`:

```bash
docker compose run --rm app psql "$(docker compose exec -T postgres bash -c 'echo "host=postgres user=solamon password=$POSTGRES_PASSWORD dbname=solamon"')" -f /app/migrations/0001_initial.sql
```

The `app` container is the FastAPI Python image. There is no spec doc for the cloud-app Dockerfile that guarantees `psql` is installed in it, that the migration files are at `/app/migrations/`, or that the Python package layout supports `python -m solamon.seed_reference_data` (line 348).

These are all build-time decisions that live in a Dockerfile that doesn't exist yet.

The simpler fix lives one container away: `docker compose exec -T postgres psql -U solamon -d solamon -f /migrations/0001_initial.sql` — but the Postgres compose definition (line 222-237) doesn't mount `./migrations/` either. Real path issue.

Fix: either mount migrations into the postgres container and exec there, or formally spec the app Dockerfile (with `psql`, the migrations dir, the Python package layout) and link from `cloud-bootstrap.md`.

## Major — gaps that will surface during deployment

### 8. `MOSQUITTO_CLOUD_PASSWORD` is `${DB_PASSWORD}` — single secret reused

Already covered in #4, but worth its own line in the punch list. Generate two distinct secrets.

### 9. Backup bucket verification + `pg_dump` cron are spec'd but not in the script

`cloud-bootstrap.md §5`: "The script also installs a daily cron at 03:00 UTC that runs `pg_dump -Fc` and uploads to S3."
`cloud-bootstrap.md §5`: "The script verifies the bucket exists; if not, exits with a hint."

Neither is in `solamon-cloud-bootstrap.sh`. No `aws s3` step, no `crontab` for `pg_dump`. Add both.

### 10. `--restore-from` skips the migration step entirely — breaks as soon as 0002 ships

`scripts/solamon-cloud-bootstrap.sh:328-336` restores via `pg_restore` and jumps directly to step 16. The restored DB is at whatever schema version the dump was at (e.g., 0001). When 0002 lands and the app expects schema 0002, the app fails to start.

Fix: after restore, replay any migrations newer than what the dump contained. Requires tracking the migration version inside the dump's `schema_migrations` table — which already exists per `data-model.md`.

### 11. Step counter mismatch in `solamon-edge-install.sh`

`scripts/solamon-edge-install.sh:38`: `TOTAL_STEPS=22`. Actual `log_step` invocations: 16. The operator sees `[16/22]` as the final step before "Edge agent ready" — confusing.

`pi-install.md §3` documents 22 numbered procedural steps; the script collapses several into single `log_step` calls. Either set `TOTAL_STEPS=16` to match the script, or split the script's log_steps to match the spec.

### 12. AWS CLI v1 from apt — deprecated for new deployments

`scripts/solamon-edge-install.sh:152`: `apt-get install -y awscli`. Pi OS (Bookworm) and Ubuntu 24.04 both ship AWS CLI v1 in the `awscli` package. AWS announced end of v1 support; new deploys should use v2.

Fix: use the official v2 ARM64 installer (`https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip`).

### 13. Secrets passed via argv leak to `ps` and shell history

`pi-install.md §2` shows the install script invocation with `--bearer-token`, `--ecr-access-key-id`, `--ecr-secret-access-key` on the command line. These are visible in `ps -ef` output to anyone with access to the Pi during the 5-10 minute install window — and may end up in `~/.bash_history` if the operator pastes the command.

For a contractor-runs-this-at-customer-site scenario, this is a credential exposure.

Fix options:
- Read secrets from a file passed as `--config /tmp/install-config.yaml` then `shred` it after.
- Or accept secrets via env vars only, with an obvious "won't be logged" note.

### 14. Tailscale `--reset` discards operator customisations on idempotent re-run

Both scripts use `tailscale up ... --reset`. The `--reset` flag clears all settings (not just authenticated state). If an operator has manually configured `--accept-routes`, `--exit-node`, or DNS settings between installs, re-running discards them.

For genuine idempotency, drop `--reset` and rely on `tailscale up` re-using the existing identity if the auth key matches.

### 15. No CI release workflow committed

`ecr-and-images.md §4` describes `.github/workflows/release.yml` in detail. No such file exists in the repo. Acceptance §10 ("A test image pushed end-to-end") depends on this CI being built. Acceptable for a spec-only phase, but should be tracked as a follow-on artifact.

### 16. `pi-ecr-puller` IAM access keys are long-lived and rotated manually

`ecr-and-images.md §6.1`: "Rotated quarterly (calendar reminder)." Manual rotation = forgotten rotation. There is no script and no Tailscale-side mechanism to push the new keys to all deployed Pis when the IAM user's credentials rotate.

For MVP single-Pi this is fine. After site #2 it's a real maintenance item.

Document: a rotation procedure that names every Pi requiring re-write of `/etc/solamon/aws-credentials`. Or move forward on the SSM hybrid plan (SOL-10).

### 17. Mosquitto password file lifecycle is unspecified

The bootstrap script creates an empty `passwords` file. `cloud-bootstrap.md §10` "first site procedure" says: "The cloud's site-add procedure regenerates `mosquitto/passwords` + `mosquitto/acls` and SIGHUPs the Mosquitto container."

But:
- "The cloud's site-add procedure" is not specified anywhere.
- `mosquitto_passwd` is the standard mechanism — does the cloud app shell into the Mosquitto container, or write the hashed entries directly?
- SIGHUP on Mosquitto re-reads ACLs but **does not** re-read passwords on all versions; depends on `auth_plugin_opt`.

Either spec the site-add path concretely (which Linear ticket covers it?) or seed the bench site's password directly in the bootstrap script and document Mosquitto reload semantics.

### 18. EC2 bootstrap key vs Tailscale SSH transition has a one-shot lock-out window

`cloud-bootstrap.md §1` step 3: "SSH to the EC2 with the bootstrap key (one-time)". After Tailscale joins (script step 5), UFW configuration (step 12) blocks port 22 from the public internet.

If Tailscale fails between step 5 success and step 12, AND the bootstrap SSH key is removed before tailnet works, the operator is locked out. The script's `set -e` means step 5 failure aborts before UFW changes — ✓ safe — but the spec should explicitly state: "do NOT remove the bootstrap SSH key from authorized_keys until you've verified Tailscale SSH works from a separate session."

### 19. CloudTrail and IAM groups described but not configured by any script

`aws-account.md §6` says enable CloudTrail. `aws-account.md §4.1-4.2` define IAM groups + service roles. None of this is automated — it's all manual AWS console work. For a one-time setup that's fine, but add a single checklist file (`aws-account-setup-checklist.md`) so a future operator can follow the steps without reading the prose.

### 20. ECR repo `solamon-cloud-postgres` is mentioned but unused

`ecr-and-images.md §1` lists it as "optional — only if we customise Postgres + Timescale image; otherwise we use the upstream `timescale/timescaledb:2.14-pg16` directly." The bootstrap script uses the upstream image. Either remove the row from the spec table or note "(MVP: not used)".

## Minor — doc consistency / polish

### 21. `AWS_ACCOUNT_ID` validation absent on the Pi

`scripts/solamon-edge-install.sh:157`: `AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)`. If the AWS CLI silently returns empty (e.g., transient API blip with `set -e` not triggering on output-only flow), `ECR_REGISTRY` becomes `.dkr.ecr.af-south-1.amazonaws.com` — invalid. The next `docker login` fails with a confusing error.

Add `[[ -n "$AWS_ACCOUNT_ID" ]] || die "AWS account discovery failed"`.

### 22. NTP sync timeout is 2 minutes — tight on marginal LTE

`scripts/solamon-edge-install.sh:103`: `for i in $(seq 1 24); ... sleep 5`. 120 seconds. On a Pi with weak LTE doing first DNS + first NTP, it's plausible to need 3+ minutes. Bump to 300 s; document.

### 23. Cron PATH default

The cron line at edge-install line 220 sets `AWS_SHARED_CREDENTIALS_FILE` and `AWS_PROFILE` but doesn't set `PATH`. Default cron PATH is `/usr/bin:/bin`. `aws` CLI on Debian: `/usr/bin/aws` ✓. `docker`: `/usr/bin/docker` (Docker apt install) ✓. ✓ Should work, but explicitly setting `PATH=/usr/bin:/usr/local/bin:/bin` is defensive.

### 24. DNS propagation wait absent before Caddy provisions

If the operator runs the bootstrap script immediately after creating the DNS A record (TTL 300), Caddy's HTTP-01 challenge can fail until DNS propagates. The 3-minute health-check window may expire before Caddy succeeds.

Add a "wait until `dig +short ${DOMAIN}` returns the EC2 IP" check before starting Caddy.

### 25. Restore branch step counter math is off by one

`scripts/solamon-cloud-bootstrap.sh:336`: `STEP_NUM=$((STEP_NUM + 1))   # skip seed step counter`. With `TOTAL_STEPS=18` regardless of branch, the restore path's displayed step counts diverge. Cosmetic.

### 26. Pi user `pi` vs `solamon` user split semantics

The Pi has user `pi` (login + docker group) and `solamon` (system user, owns `/etc/solamon`). The agent container runs as root inside (`network_mode: host`, no `user:` directive). If the agent's Dockerfile later switches to non-root, the host's `chmod 700 /etc/solamon` blocks the in-container read.

Document the assumed in-container UID, or set `chmod 750` with group ownership to a known UID.

### 27. `solamon-edge-install.sh` defaults `DEVICE_HOST=192.168.1.254` only

The Acuvim factory default. Sites with non-default LAN ranges (`10.0.0.x`, `172.16.x.x`) fail unless `--device-host` is overridden. Document a "before you run" checklist that says "confirm the Acuvim's IP first; pass `--device-host` if not 192.168.1.254".

### 28. `update.sh` and `decommission.sh` referenced but not committed

`pi-install.md §6, §7`. Both are described as future polish. Acceptable in spec phase; flag as a tracking item for SOL-21 follow-on.

### 29. Tailscale free tier limits (100 devices, 3 users)

`tailscale.md §1`. Fine for MVP single-site. After the 4th human user (subcontractor commissioning?), you hit the user limit. Note in §1.

### 30. AAAA record explicitly empty

`caddy-and-dns.md §2`: no IPv6. ✓ Acceptable. Worth a note that if a site's LTE provider becomes IPv6-only (rare but possible in some regions), the Pi can't reach the cloud.

### 31. `tailscale ip --4` can return empty momentarily after `tailscale up`

Both scripts call this after a `sleep 5`. Usually fine; on slow tailnet announce, may return empty and break the final "Pi tailnet IP" log line. Cosmetic.

### 32. CloudTrail config not tied to S3 bucket creation

`aws-account.md §6` says "configure it to also write to an S3 bucket (`solamon-cloudtrail-prod`)". The S3 bucket isn't created by any script — manual setup. Add to the AWS-account checklist.

### 33. `random_secret()` truncates to 24 chars by default

`scripts/solamon-cloud-bootstrap.sh:56-58`: produces an alphanumeric string. ✓ Safe for the JWT_SECRET / DB_PASSWORD / etc. (32 chars for some, 24 for others). 24 chars from `[a-zA-Z0-9]` ≈ 143 bits of entropy ✓ adequate.

### 34. `--admin-email` defaults to `admin@solamon` (not a real domain)

`scripts/solamon-cloud-bootstrap.sh:28`: `ADMIN_EMAIL="admin@solamon"`. NextAuth's email validation may reject this (no TLD). Either default to `admin@${DOMAIN}` or leave undefined and require the operator to pass `--admin-email`.

### 35. No `ufw` firewall on the Pi

The cloud's `solamon-cloud-bootstrap.sh` configures UFW; `solamon-edge-install.sh` does not. Pi's port 22 (OpenSSH) is open from the LAN. With `network_mode: host` on the agent, any container-listening port would also be exposed.

For MVP the Pi sits on a (presumably) private site LAN. Consider adding a single `ufw allow in on tailscale0 && ufw deny incoming && ufw allow ssh` block, or document the threat model explicitly.

### 36. OpenSSH key fallback is described in `tailscale.md §6` but not in the install script

The "belt-and-braces" SSH-key fallback is mentioned but the install script doesn't add anyone's public key to `~pi/.ssh/authorized_keys`. If Tailscale SSH ever fails, recovery requires either physical access or a pre-existing Pi Imager-set password.

### 37. EC2 IAM instance profile assumed but not created by script

`cloud-bootstrap.md §1` step 1: "IAM instance profile: `EC2InstanceRole`". Created manually per `aws-account.md §4.2`. Not in any script. Operator must do this in the AWS console before `solamon-cloud-bootstrap.sh` runs — easy to miss. Add a pre-flight check that tests `aws ecr get-login-password` and fails fast with a clear message if the instance profile is missing.

## Summary

The infrastructure spec group is the only one with executable artifacts. That makes the bugs more concrete — and most of them concentrate around the cloud bootstrap script's TLS-cert-share between Caddy and Mosquitto.

**The Mosquitto path is broken in five compounding ways** (#1-#5): the volume points to the wrong storage layer, the filenames don't match, `cafile` enforces unintended mTLS, the password file is empty, and its permissions are wrong. Even fixing one of these doesn't get the bootstrap script to a green health check; the spec needs a coherent revision of the entire cert-and-auth handshake. The simplest path is the MVP-pragmatic certbot quarterly cron suggested in `caddy-and-dns.md §6` itself — abandon the auto-share, populate the password file via `mosquitto_passwd` at site-add time.

**#7 (migration via app container)** is a cross-spec gap: the app Dockerfile spec doesn't exist, but the bootstrap script makes assumptions about its contents. Either add a minimal Dockerfile spec or move the migration to the Postgres container.

**#10 (restore-from skips migration replay)** is the silent landmine: as soon as 0002 ships, the disaster-recovery path fails to bring the cloud back up.

**#13 (secrets in argv)** is a real-world security gap that the contractor-on-site scenario in the runbook makes plausible.

Everything else — step counters, AWS CLI v1, missing CI workflow, missing decommission scripts — is tractable polish or follow-on tickets. The Mosquitto cert mess and the migration path are the items to fix before bench Day 1.
